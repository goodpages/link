解析器生成器——Lemon

Lemon是一个使用LALR算法的parser（解析器）生成器，它输出C代码。Lemon的用途类似于"yacc"或者"gnu bison"，但Lemon不仅仅是又一个yacc那么简单。它使用了和yacc有所区别的语法，以帮助使用者避免常见错误。Lemon输出的解析引擎比yacc和bison的速度更快，同时还是可重入和线程安全的。Lemon还可以帮助你避免资源泄露。这一切使得Lemon比yacc更适合于长时间运行的程序例如图形界面或者嵌入式控制器。

1.操作指南

Lemon的主要目标是把针对某特定语言的上下文无关文法翻译为对此语言的解析器实现。Lemon接受两个输入文件：

  • 语法定义
  • 解析器模版

通常，使用者只需要提供语法定义，Lemon会使用自带的默认模版，它对于大多数应用都是合适的。

随命令行参数的不同，Lemon可能输出1到3个文件：
  • 实现解析器的C文件。
  • 一个头文件，在其中定义每个非终结符为一个整数常量
  • 一个报告文件，描述了解析器中自动机的状态。
默认情况下，三个文件都会生成，可以使用"-m"选项来抑制头文件的产生，用"-q"选项抑制报告文件产生。

语法定义文件使用".y"后缀，接下来的例子中，我们使用"gram.y"作为语法文件的名字。这样，调用Lemon的命令行就是

   lemon gram.y
运行命令后，将看到产生了三个文件"gram.c"，"gram.h" 和"gram.out“。

1.1 命令行参数

命令行参数可以控制Lemon的行为，要获得完整的可用命令行参数列表外加简短说明，可以： 
   lemon -?
在本文档写作时，下面的选项是可用的：
  • -b
  • -c
  • -g
  • -m
  • -q
  • -s
  • -x

"-b"选项将过滤报告文件中的输出，使其仅仅列出每个解析状态的基本情况。"-c"选项抑制动作表的压缩，这将导致更大更慢的解析器，同时加快Lemon检查语法错误的速度。"-g"选项将抑制所有输出文件，转而把所有语法规则连同注释打印到标准输出上，动作和其他部分将被过滤，这样可以快速的得到对语法概述。"-m"选项是的产生的C文件与"makeheaders"程序兼容，你可以用makeheaders来从C文件中自动抽取头文件。当使用"-m"选项时，头文件不会产生。"-q"选项抑制产生报告文件。如果使用"-s"选项，Lemon会在标准输出打印出解析器的统计信息，类似于：

Parser statistics: 74 terminals, 70 nonterminals, 179 rules 340 states, 2026 parser table entries, 0 conflicts Finally,

"-x"选项使得Lemon仅仅打印版本号与版权信息后直接退出，不读取语法文件。

1.2 解析器接口

Lemon并不生成完整的可执行程序，而仅仅生成一族实现解析器的子例程。程序员可以调用它们来实现完整的程序。
在使用Lemon生成的解析器之前，必须先创建一个解析器实例，就像下面这样： 
   void *pParser = ParseAlloc( malloc );
ParseAlloc()函数分配并初始化一个新解析器，并返回指向它的指针。而解析器实际的数据结构对用户是透明的——内部结构不可见。因此，ParseAlloc()返回的是void *指针而不是指向某种特定结构的指针。仅有的一个参数是用来分配内存的接口，其原型同标准库的malloc()函数。

在不再需要此解析器时，可以回首分配的内存： 
   ParseFree(pParser, free);
第一个参数就是从ParseAlloc()返回的透明指针，而第二个参数则是释放内存的函数指针。

在解析器实例化后，就可以向其中输入记号（终结符）流进行解析了。下面的函数每次接受一个非终结符。
   Parse(pParser, hTokenID, sTokenData, pArg);
第一个参数为解析器实例，第二个参数是一个 正整数，代表非终结符类型。这些类型都用#define预处理指令定义在Lemon输出的头文件"gram.h"中。如果第二个实际参数值为0，则表明输入记号流终结（EOF）。第三个参数代表非终结符的语义值，在默认情况下，第三个参数的类型是int，但语法文件中经常需要重定义这个类型为某种结构。

Parse()函数可以有三个或四个参数，这取决于语法定义。如果语法需要，Parse()函数可以有任意类型（由用户定义）的第四个参数，解析器并不处理这个参数，而仅仅把它传递给用户提供的动作。这可以方便的被用来传递参数给解析器内部，而不需要动用全局变量。

一段典型的使用Lemon生成的解析器的代码看上去像这样：

   01 ParseTree *ParseFile(const char *zFilename)  {    02    Tokenizer *pTokenizer;    03    void *pParser;    04    Token sToken;    05    int hTokenId;    06    ParserState sState;    07    08    pTokenizer = TokenizerCreate(zFilename);    09    pParser = ParseAlloc( malloc );    10    InitParserState(&sState);    11    while( GetNextToken(pTokenizer, &hTokenId, &sToken) ){    12       Parse(pParser, hTokenId, sToken, &sState);    13    }    14    Parse(pParser, 0, sToken, &sState);    15    ParseFree(pParser, free );    16    TokenizerFree(pTokenizer);    17    return sState.treeRoot;    18 }
这个例子展示了一个用户例程，它可以解析文件并返回生成的解析树的指针（我们省略了所有的错误处理）。我们假设有某种类型的词法分析器在第八行用TokenizerCreate()创建，并在16行销毁。11到13行的主循环完成了大部分的工作。sToken可能是包含了复杂的语义值的结构类型。

这个例子还假设有一个ParserState的实例，其中维护着一次特定的解析所需要的状态信息，它在第6行创建，第10行初始化。并在每次调用Parse函数时作为可选的第4个参数被传递进来。在本例中，我们假设ParserState结构的分量treeRoot是指向语法树的指针。

这个例子与Lemon的主要代码如下：

   ParseFile(){       pParser = ParseAlloc( malloc );       while( GetNextToken(pTokenizer,&hTokenId, &sToken) ){          Parse(pParser, hTokenId, sToken);       }       Parse(pParser, 0, sToken);       ParseFree(pParser, free );    }
通常来说，使用Lemon产生的解析器的程序首先创建/实例化解析器，然后向它传递记号流，当到达文件尾时，就把第2个参数设为0调用Parse，最后，用ParseFree()释放内存。

在继续深入之前，还有一个接口函数值得注意，ParseTrace()函数可以用来从解析器中产生调试输出，其原型如下： 
   ParseTrace(FILE *stream, char *zPrefix);
在此例程被调用后，每次当解析器状态改变或者调用动作时，都会向文件流stream中，以zPrefix作为前缀输出单行调试信息。用NULL(0)作为第一个参数再次调用此函数将关闭调试i信息输出。

1.3与Yacc和Bison的区别

先前使用过yacc或bison的程序员可能会注意到Lemon与它们一些重要的不同点：

  • 在yacc中，由prser调用lexer，而在Lemon中，由Lexer调用parsr。
  • Lemon不使用全局变量在lexer和parser之间传递信息。
  • Lemon允许多个解析器实例同时运行（可重入），而yacc和bison不能。
这些不同可能在开始时对那些有yacc经验的程序员带来不适，但在使用Lemon几年之后，我坚信Lemon的方法更好。

2.输入文件语法

语法文件不光定义了解析器使用的语法，还指定了当规则匹配时执行的动作，和其他附加信息。使用Lemon的主要过过程就是打磨语法文件。

不像Yacc，Lemon的语法在大多数情况下是自由格式的，也即不存在段/分割符。声明可以出现在文件中的任何地方。Lemon忽略空白（除非需要空白来分隔记号）。Lemon接受与C和C++相同的注释语法。

2.1 终结符与非终结符

一个终结符（记号）以大写字母开头的一串【字母|数字|下划线】，但惯例上使用全大写字母。一个非终结符，则是以小写字母开头的 一串【字母|数字|下划线】，惯例上使用全小写。

在Lemon中，终结符和非终结符不需要在一个单独的段中声明，lemon能够从语法规则中自动产生所有记号的列表，区别非终结符和终结符的依据即是上面的命名规范。

Yacc和bison允许终结符为单独的由单引号括住的字符，例如')'和'$'，Lemon不接受这样的终结符，在Lemon中，所有的记号，都是字母数字混合串。

2.2语法规则

Lemon语法文件的主要部分是一系列的语法规则。每个语法规则由一个非终结符后跟"::="，接着是（非）终结符的列表。产生式右侧可以为空。产生式可以以任何顺序书写，但第一条被当作开始符号。（除非用%start指令另外指定）。典型的语法规则就像下面这样：

  expr ::= expr PLUS expr.   expr ::= expr TIMES expr.   expr ::= LPAREN expr RPAREN.   expr ::= VALUE.

There is one non-terminal in this example, ``expr'', and five terminal symbols or tokens: ``PLUS'', ``TIMES'', ``LPAREN'', ``RPAREN'' and ``VALUE''.

上面的规则中有一个非终结符，"expr"，和五个终结符（记号）， ``PLUS'', ``TIMES'', ``LPAREN'', ``RPAREN'' 和`VALUE''。

类似yacc，Lemon允许在规则后面书写C代码块作为语法规则规约时执行的动作。在Lemon中，动作是由花括号包裹的C代码组成，紧跟在由英文句号结束的产生式后面，例如： 
  expr ::= expr PLUS expr.   { printf("Doing an addition...\n"); }
为了对语义值进行操作，语义动作必须和语法规则联系起来。在yacc和bison中，用'$$'标志产生式左侧非终结符的语义值，而用符号"$1"，"$2"等来代表产生式右侧的终结符/非终结符的语义值。这个方法广为人知，但也很容易导致错误。最常见的就是数错产生式右侧符号的个数。

Lemon不使用这种Shell风格的变量表示方法，而是在产生式中使用显式的符号名字，然后在动作部分可以用这些名字应用对应的语义值。在yacc或bison中，可能写成这样：

  expr -> expr PLUS expr  { $$ = $1 + $3; };
但在Lemon中，同样的规则则是： 
 expr(A) ::= expr(B) PLUS expr(C).  { A = B+C; }
在Lemon规则中，任何在跟在语法规则符号之后，由括号包裹的符号自动成为语义值占位符，占位符可以在C代码中被引用来表示其语义值。

Lemon的方法在很多方面优于yacc; 第一，就像之前提到的，Lemon的方法避免了对语法符号进行记数; 第二，如果在语法规则中指定了占位符，但在规约动作中没有引用它，那么Lemon会提示出错，例如：

  expr(A) ::= expr(B) PLUS expr(C).  { A = B; }

上面的规则将产生错误，因为占位符C在规约动作中没用到。同时这种记法还可以方便的在语义动作中回收相关的内存资源。

2.3优先级规则

Lemon使用和yacc/bison相同的方法来解决二义性问题。一个移入—规约冲突优先移入，而规约—规约冲突则优先规约在语法文件中先列出的语法规则。

类似yacc和bison，Lemon允许使用优先级规则来控制对解析冲突的处理方法。用来对终结符赋优先级值的指令包括"%left"，"%right"和"%nonassoc"。在文件较上方的指令指定的终结符比下方的优先级低。

   %left AND.    %left OR.    %nonassoc EQ NE GT GE LT LE.    %left PLUS MINUS.    %left TIMES DIVIDE MOD.    %right EXP NOT.

在上面的指令序列中，AND运算符的优先级最低，OR比AND高……因此，语法会解决二义性的表达式：

     a AND b OR c

像这样：
  a AND (b OR c).
结合性（left,right或者nonassoc）用来决定有限级相同时的二义性问题，AND在例子中是左结合的 
     a AND b AND c
等价于 
     (a AND b) AND c.
而EXP运算符是右结合的，因此： 
     a EXP b EXP c

等价
     a EXP (b EXP c).
nonassoc指令用于指出无结合性的运算符，例如 
     a EQ b EQ c

将产生错误。

产生式之间的优先级取决于产生式右侧最左边的已经定义了优先级的终结符/。这可能是你想要的，但在某些情况下，当你需要调整产生式优先级时，像下面这样，在规则之后，C代码之前放置：

   expr = MINUS expr.  [NOT]
这个产生式的优先级将等于NOT符号，而不是默认的减号(MINUS)。

现在我们知道了优先级如何被赋予终结符和单独的产生式，我们现在可以精确的解释Lemon如何解决解析中的冲突。移入—规约冲突按照下面的规则解决：
  • 将被移入的记号没有定义优先级信息，那么优先移入，但同时警告一次冲突。
  • 将移入的记号优先级高于规约的产生式优先级，那么优先移入，不产生冲突。
  • 将移入的记号优先级低于规约的产生式优先级，那么优先规约，不产生冲突。
  • 优先级相同，将移入的记号右结合的，那么优先移入，不产生冲突。
  • 优先级相同，将移入的记号左结合的，那么优先规约，不产生冲突。
  • 否则，优先移入并报告冲突。
规约—规约冲突按照下面的方法解决：
  • 如果两个将规约的产生式都无优先级信息，那么优先规约先出现的产生式并报告冲突。
  • 否则，如果优先级不同，先规约优先级高的。
  • 再否则，规约先出现的规则，并报告冲突。

2.4 特殊指令

Lemon接受的输入语法包括语法规则和特殊指令。我们已经解释了语法规则，下面介绍特殊指令。

Lemon中的指令可以按照任意顺序出现，你可以把它们放置在语法规则之前，或者之后，甚至中间，效果是相同的。赋优先级值的指令之间的相对顺序是有意义的，但其他的指令顺序则无所谓。

%destructor

%destructor指令用来为非终结符指定析构函数(类似的%token_destructor指令则为终结符指定析构函数)。

当非终结符被弹出栈时，包括下面这些情况，将执行析构函数：

  • 当产生式规约，同时产生式右侧某非终结符的语义值在C代码中未被引用。
  • 错误处理时，堆栈中不断弹出记号。
  • 执行ParseFree()函数。
析构函数可以对非终结符的语义值做任何事情，但设计上，一般只用来释放占用的内存等资源，看下面的例子： 
   %type nt {void*}    %destructor nt { free($$); }    nt(A) ::= ID NUM.   { A = malloc( 100 ); }
这个例子看起来有点假，但还是演示了析构函数的工作方法。例中，终结符"nt"的语义值类型为"void *"，当以nt为LHS的产生式规约时，它为nt的语义值A分配空间。接下来，当nt从栈中弹出时，析构函数被调用来释放内存，这避免了内存泄露（注意到$$被nt的语义值所替代）。

注意，无论何时，只要非终结符出栈且其值没有在相关动作代码中引用，就会调用析构函数。如果C代码中用到了其语义值，则Lemon假设C代码会释放其空间。一般来说，这种情况下它的值会被用来构建更大的结构（例如AST）因此不能被释放。

合理的使用析构函数，可以用lemon构建一个长时间运行的程序，类似于GUI，而不会资源泄露。用yacc或bison达成同样的目标会更困难一些。

%extra_argument

这个选项命令Lemon在生成的解析器中，接受可选的第四个参数。例如，如果语法文件包含： 
    %extra_argument { MyStruct *pAbc }

那么生成的Parse()函数会有一个类型为"MyStruct *"的第四个参数pAbc，所有的动作例程都可以访问pAbc，其值为最近一次调用Parse时传递的值。

%include

%include指令后花括号内包裹的代码被包含在生成代码的最上方，一般用来包含头文件。多个%include块的顺序被保留。 
   %include {#include <unistd.h>}

%left

%left指令（和%right以及%nonassoc）用来声明终结符的优先级。每个名字出现在%left指令之后，和下一个英文句点之前的终结符名字都有同样的左结合优先级。而下面出现的%left指令具有更高的优先级。例如：
   %left AND.    %left OR.    %nonassoc EQ NE GT GE LT LE.    %left PLUS MINUS.    %left TIMES DIVIDE MOD.    %right EXP NOT.

注意英文句点结束一个%left，%right或%nonassoc指令。

LALR(1)语法在使用过多的右结合运算符时，可能会不停的移入，出现很大的堆栈。因此，推荐在可能的时候，尽可能多使用左结合运算符。

%name

默认情况下，Lemon生成的函数都有字符串前缀"Parse"，可以用%name指令修改这个前缀，例如： 
   %name Abcde

Lemon生成的函数名将是：

  • AbcdeAlloc(),
  • AbcdeFree(),
  • AbcdeTrace(), 和
  • Abcde().
这个功能让你能够用Lemon生成几个不同的Parser并链接到同一个程序中。

%nonassoc

参见优先级或者%left。

%parse_accept

%parse_accpet指令指定的代码块将在解析器每次接受输入串时执行。所谓接受输入串，就是解析器处理了所有的记号而没有发现错误，例如：: 
   %parse_accept {       printf("parsing complete!\n");    }

%parse_failure

这条指令指定的代码块只有在解析器不能完成工作时执行，即解析器使用通常的错误回复策略不能解决输入中出现的错误时，才被执行。 
   %parse_failure {      fprintf(stderr,"Giving up.  Parser is hopelessly lost...\n");    }

%right

参见优先级和%left部分

%stack_overflow

指定当内部栈溢出时执行的动作。一般仅仅打印一条错误信息。在堆栈溢出后，解析器不能继续，必须被重置(reset)。 
   %stack_overflow {      fprintf(stderr,"Giving up.  Parser stack overflow\n");    }

通过避免使用右递归产生式和右结合运算符，而使用嘴递归和左结合，可以避免发生溢出，例如：

   list ::= list element.      // left-recursion.  Good!    list ::= .
而不是： 
   list ::= element list.      // right-recursion.  Bad!    list ::= .

%stack_size

如果使用左递归仍然不能解决溢出，就不得不考虑增加堆栈大小了。stack_size指令后跟一个正整数。Lemon默认的栈大小是100。 
   %stack_size 2000

%start_symbol

默认的，开始符号是列出的第一个产生式的LHS非终结符，用%start_symbol指令可以另外指定。

   %start_symbol  prog

%token_destructor

类似于%destructor对非终结符所做的那样。

不同于非终结符，它们可能各有个的数据类型。所有终结符共享同一数据类型（用%token_type指令定义）。所以它们使用同一个析构函数而无需分别指定。

%token_prefix

Lemon生成的头文件中，会为终结符定义整形常量。如果需要，Lemon可以为每个#define添加一个前缀。

默认的输出：

    #define AND              1     #define MINUS            2     #define OR               3     #define PLUS             4
如果添加下面的指令： 
    %token_prefix    TOKEN_

Lemon的输出就像这样：
    #define TOKEN_AND        1     #define TOKEN_MINUS      2     #define TOKEN_OR         3     #define TOKEN_PLUS       4

%token_type 和 %type

这两条指令用于为解析器堆栈中的终结符和非终结符指定数据类型。所有的终结符数据类型一致，也就是Parse()函数的第三个参数。典型的，终结符的类型可以是指向某个结构的指针。 
   %token_type    {Token*}

如果没有为终结符指定数据类型，默认值为``int''。

不同的非终结符一般有自己的数据类型，典型的非终结符数据类型是指向包含所有信息的解析树树根的指针，例如：

   %type   expr  {Expr*}

解析器堆栈里的每一项实际上是一个union的实例，相容于所有的非终结符和终结符数据类型。Lemon会根据当前的语法符号类型自动使用正确的分量。但使用者需要注意：union的大小取决于最大的数据类型，假如有单独的非终结符占据了1k内存，那么100个项大小的栈就占据了100k。

%fallback

%fallback ID X Y Z.

ID作为非终结符X、Y和Z的备选，如果输入需要移入一个非终结符X或Y或Z，而输入的不是，那么下一步parser可以接受ID作为替代，如果下一个输入不是ID才会引发错误。

2.5错误恢复

在纠结了好几年之后，我发现yacc使用的错误恢复策略是最好的，因此这也是Lemon的方法。

当一个Lemon产生的parser遇到语法错误时，首先调用%syntax_error指令指定的代码，如果存在的话。接下来，进入错误恢复过程：不断的从栈内弹出元素，直到进入一个状态，此状态允许移入一个特殊的非终结符"error"，然后它移入"error"非终结符并继续解析。但%syntax_error例程在至少三个新非终结符被成功移入前不会被调用。

如果parser弹空了自己的堆栈，仍然无法移入error符号，那么就调用%parse_failed例程，并重置自己到开始时的状态（在开始解析新文件时的状态）。在输入的一开始就遇到语法错误时就会使用这种策略。当然，如果栈中没有error非终结符实例的话。
