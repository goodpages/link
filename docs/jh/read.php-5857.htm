  1. 为什么CClayer的init()和onEnter()方法有时不被执行

    init方法：定义一个Layer类时，好的习惯是init()方法和CREATE_FUNC(Layer类名);都要定义，看CREATE_FUNC()宏定义就可以明白，CREATE_FUNC是先new出这个Layer类再调用了init方法，所以我们Layer::create()时就会调用init()方法，而只是单纯的new就不会调用

    onEnter()方法不执行：这个经常遇到比较头疼，看官方的HelloWorld例子就可以知道：

    在bool AppDelegate::applicationDidFinishLaunching()方法中，有这两句是加载场景的

    CCScene *pScene = HelloWorld::scene();
    CCDirector::sharedDirector()->runWithScene(pScene); //步骤2

    而在HelloWorld::scene()中，代码是这么写的

       // 'scene' is an autorelease object
        CCScene *scene = CCScene::create();
        // 'layer' is an autorelease object
        HelloWorld *layer = HelloWorld::create();

        // add layer as a child to scene
        scene->addChild(layer);  //   步骤1

        return scene;
        这里就能明白了，要想执行layer的onEnter方法，必须有两个步骤，一是生成scene后要把这个layer给addChild，然后再run这个scene，使这个layer成为scene的孩子，又使这个scene成为活动场景，就会执行onEnter方法，具体原理看源代码可知

  2. 内存管理问题，什么时候需要调用retain？

    大多数情况下，我们new出的资源只要autorelease就行了，但有时却不行，new出的东西莫名其妙的被销毁了

    见下例：

    CCLayer* CreateLayer(int index) //这里返回一个layer

    在init方法里，我们  m_pSecondLayer = CreateLayer(2);  //生成一个后备layer

    然后在退出事件里 如onExit

    m_pSecondLayer->onExit();  //执行到这里就报错了，因为m_pSecnodLayer内存已经被回收，但不是空指针

    这就很奇怪，为什么以前用没问题这里却出问题了呢？原因在于m_pSecondLayer生成后无人认领，所以当退出当前模块后内存就会被cocos2d的内存管理池回收了，所以我们不想被回收需要在layer被create代码后立即XXXlayer->retain();而官方的例子没问题是因为他们的layer生成后立即addChild到场景中，就是被当前场景认领了，所以不会被回收，只有当前场景也被退出删除了场景的子结点包括那个layer才被回收

    总结：new出一个layer或一个node如果我们没有addChild到其他父结点上时，我们需要retain它

  3. CCNode执行scheduleUpdate()后不执行update方法

    scheduleUpdate()是执行了，但update()方法并没有被调用，从网上搜下，也并没有找到好的解决方法，最后，在热心网友kaka和你用温暖的目光(非常感谢两位)的帮助下，终于找到解决办法了，办法很简单，就是用下面这句话替换scheduleUpdate()就行啦：

    CCDirector::sharedDirector()->getScheduler()->scheduleUpdateForTarget(this,0,false);

    或者：

     CCDirector::sharedDirector()->getScheduler()->scheduleSelector(schedule_selector(NewGame::update),this,0.1,false);

    释放用：

    CCDirector::sharedDirector()->getScheduler()->unscheduleUpdateForTarget(this);

    4. CCDirector::sharedDirector()->getScheduler()->unscheduleSelector(SEL_SCHEDULE(&Potentiometer::valueChange), this);bw
    附：上面的调用太过吓人，有一种简单的指定间隔时间执行某个update的简单方法

        在init里 ：

    schedule(schedule_selector(BattleController::updateBattle), 5); //每5秒执行一次updateBattle函数

    在退出或不想再更新时:

   unschedule(schedule_selector(BattleController::updateBattle));

   updateBattle方法的声明是这样的：void BattleController::updateBattle(float dt)

   定时器写法(固定时间后执行某个函数，且只执行一次)：

  scheduleOnce(schedule_selector(CUIHello::GotoLogin), CALLBACK_DELAY);


4. 动作Action在update中每帧调用产生叠加的问题

   我们在想移动sprite坐标，一般调用CCMoveBy或CCMoveTo,事先根据物体要移动的目标算出距离再除以时间，可得CCMoveTo的时间参数，可是经常遇到要跟踪一个object的移动 ，如跟踪子弹，那我们可以在update里根据变化的目标object坐标一直调用CCMoveTo吗？答案是不可以直接调用，因为如果你用scheduleUpdate来使用update，那update是每帧都调用的，会产生叠加动作问题，CCMoveTo的效果就不可预估了，当然你可以直接用setPosition来解决，但由于update是每帧都执行，有的机器帧率快有的慢，这又会产生显而易见的问题

从上可知，每帧来修定位置的老传统是不可行了，所以cocos2d-x引入了CCMoveTo动作，使用一定时间来执行来避免机器性能不同造成时快时慢问题，但动作又不能产生叠加，怎么办呢？答案还是可以在update里执行，但是要指定update间隔调用时间，即每隔N秒(一般是0.3，0.5秒)执行，如果你直接用scheduleUpdate，那update就每帧执行，可以使用在3里我们提到的函数:

CCDirector::sharedDirector()->getScheduler()->scheduleSelector(schedule_selector(NewGame::update),this,0.1,false); //0.1就是间隔执行时间

5. 如何继承的类中采用组合CCSprite方式，能调用父类的setPosition来调整CCSprite位置

常见的一个类有显示用的CCSprite，还有其他一些数据变量，如血量，MP等，这个类要继承于CCNode,然后CCSprite是采用组合方式使它成为类的一个成员变量，然后在这个类的初始化时CCSprite生成后就addChild(CCSprite)加到父类的CCNode中,这样调用setPostion就相当于调整这个CCSprite的位置了

6. CCScale9Sprite 第2个和第3个参数什么意思？


CCRect rectFull = CCRectMake(0, 0, 10, 10);

CCRect rectCapInsets = CCRectMake(4, 4, 2, 2);

CCScale9Sprite * scalesprite = CCScale9Sprite::create("box.png", rectFull, rectCapInsets);

rectFull是整个9宫大小，rectCapInsets里九宫里面那个格子，就是可以向里加东西的格子位置与大小

加入图片是圆角图片，圆角半径为3，图片大小为16x16，则rectInsets最好设置为(3,3,10,10); 其中，3,3是中间区域的起点，10,10是中间区域的宽高,10是16-2*3的结果.

示例博客：http://blog.itpub.net/14466241/viewspace-762754

7.removeFromParentAndCleanup(true)后，原UI上的按钮仍能响应

从父结点上删除一个孩子，如从父UI上关闭一个子界面，常用的就是这个函数，但调用后发现有按钮响应的地方还能响应，原因就是此子界面的onExit()没有调用CCLayer::onExit().
