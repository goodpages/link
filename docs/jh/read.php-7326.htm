为了降低成本，任天堂没有采用当时已有的16位的CPU，而最终采用了一个过时的、8位的CPU（所以有时NES在国内也被称为“8位机”，区别于后来的“16位机”）。这个CPU的型号为2A03，是6502的一个变种。由于2A03没有图像处理功能，所以任天堂不得不采用了一块单独的PPU（Picture Processing Unit）来进行图像计算和输出。

 

无论是CPU和PPU都有自己的RAM，而游戏存储于卡带中的ROM中，当卡带被插到游戏机上时，CPU就能访问到游戏ROM了。当然有些游戏也有自己的RAM，用于保存用户存档，由电池来供电。

 

在本篇中将主要针对CPU的RAM也就是主存进行介绍。

 

 

 

1.CPU概述

NES用的CPU为2A03，由6502改进而来。两者的共同点在于：

1)    指令集相同

2)    都为小端(little-endian)

 

两者的差别主要在：

1) 2A03多一个pAPU(pseudo-Audio Processing Unit)，用于处理声音

2) 2A03缺少BCD（Binary Coded Decimal）模式。

*BCD模式，即用二进制来模拟十进制数，具体来说，就是用4个二进制bit表示十进制中的一位。比如123，用二进制表示为01111011,而用BCD表示为0001,00010,0011（逗号只是为了视觉上更清晰）。

 

 

 

 

 

2.Architecture & Memory Map

内存主要分为这三部分：ROM, RAM, I/O内存镜像。同样有三根总线，地址总线、数据总线和控制总线，地址总线和数据总线顾名思义，其中控制总线用来告诉其它部件这是个读请求还是写请求。ROM是在卡带中，需要通过MMC来进行访问，这样bank switching才有可能（具体见下一段）。而I/O的内存镜像使得cpu可以和其它部件例如ppu或者输入设备进行沟通。

其中地址总线为16位，即可以支持64kb(2^16)的内存空间访问,从$0000到$FFFF。而控制总线和数据总线为8位。

 

MMC(Memory Management Chip)作为一个独立的硬件存在于卡带中。但是并不是每个游戏卡带都包含MMC，比如超级玛丽就没有MMC，不过几乎任何比超级玛丽后推出的游戏都会包含MMC。那么它究竟起到什么作用呢？简单来说，NES系统需要将游戏ROM加载到主存，早期的NES系统的主存大小确实能满足读取ROM的需求，但是不幸的是后来游戏越做越大，NES已经无法一次性地把所有ROM装载进主存。于是就用了一个妥协的设计，MMC就出现了。它的作用有点类似于现代OS中“虚拟内存”的运作方式，当NES需要加载一个ROM Bank时（bank是rom的单位，一个bank是16kb或8kb，取决于rom的类型。PRG-ROM一个bank是16字节），而这个bank当前并没有被读进主存中，这时候就会经由MMC把需要的bank读进来置换现有的一个bank。更具体说，rom会被加载到内存中$8000-$9FFF的区域，如果当前加载进的是bank 1，那么$8000就会是bank 1的第一个字节；如果当前加载进的是bank F，那么$8000就会是bank F的第一个字节。

 

下面是NES的memory map。左边是总称，右边是更进一步的划分。

上图中，最底下的$0000-$00FF被称为zero page，一般page指的是内存中的分页，一个page一般是4k字节，这里zero page刚好内存中的最低的4个字节，也就是page id 0，故名zero page。为何要单拿它出来讲呢？主要是它有个特点是它的地址只要1个字节就能表示，所以如果对该区域进行读写的话，从效率出发，就不用再传16位的地址了，只要传8位就行了，这样指令长度中的操作数的长度部分就能得到压缩，提高了效率。因此它有自己单独的寻址模式和对应指令系统。具体的在后面寻址模式部分会细讲。

 

NES的内存中，有一个特点叫做mirrored memory（在上一篇文章中已经提到过了），比如从$0000到$07FF的空间会被镜像地复制三份放到从$0800-$1FFF当中去。这意味着，对于任何写入$0000的数据，它应当被同时写到$0800, $1000以及$1800中。而$2000-$2007这8个字节的I/O寄存器也会被复制N份填满$2008到$3FFF的地方。

$6000-$7FFF的空间用于存放SRAM(Save RAM)，它用于访问卡带中的RAM区域，在保存游戏时用。

从$8000往上存着卡带的PRG-ROM(Program ROM)。游戏卡带中的PRG-ROM的大小用bank(16kb)去衡量。如果游戏仅有1个bank大小，它会被同时复制到$8000和$C000.这是为了保证中断处理函数向量表（类似现代os里的idt，后面还会讲到）能被读取到正确的地方。如果游戏有2个bank大小，它的第一个bank会复制到$8000,而第二个bank会被复制到$C000。对于大小超过2个bank的游戏ROM,就需要根据类型具体对待了。如果你稍微看过模拟器源代码的话，应该会发现很多mapper文件，针对的就是各种不同类型的ROM的mapping情况。常见的mapper的运作方式以及对应的游戏会在之后的文章中分析。

 

衍生开去，关于金手指，可以想到的是它的工作方式其实就是锁定内存中的区域。比如一个游戏把生命值锁储存在内存的$1000中，那么锁定这块内存中的值为一个固定值，比如5，就可以给玩家无限的生命。

 

模拟器一般都会用到大量的单字节、双字节、四字节，为了便于简写以及平台一致性，一般都会用typedef来定义一些常用的字节大小。例如以下定义了Byte、Word、DWord(Double-Word)分别为无符号的1\2\4字节。

typedef unsigned char  Byte; typedef unsigned short Word; typedef unsigned int   DWord;

Memory的大小我们直到是0x10000，那么memory就可以大概定义为：

const size_t CPU_MEMORY_SIZE = 0x10000; Byte memory_[CPU_MEMORY_SIZE];

你可以依照个人语言习惯把它放在全局变量、CPU class的局部变量，或者单独拿出来做个memory的类都可以，怎么习惯怎么来。

 

 

3.寄存器

6502的寄存器较少，只有3个通用寄存器和3个特殊功能寄存器。除了PC寄存器为16位外，其他寄存器长度都为8位。

3个通用寄存器：

A(Accumulator)

X(Index Register)

Y(Index Register)

3个特殊功能寄存器：

PC(Program Counter)  - 16位

SP(Stack Pointer)

P(Status Register) *有时也简写为S

 

A寄存器：

用于存储算数运算、逻辑运算结果，也可以被用来存储从内存中读到的值。

 

X寄存器:

常被用来当做计数器或者在一些寻址方式下作为地址偏移值(offset)。也可被用来存储内存中的值，或者作为读写SP的中间值。

 

Y寄存器:

与X类似，也可作为计数器或者偏移值。但是不能被用作SP读写的中间值。

 

PC寄存器：

保存的是下一条指令的地址。除了按部就班地一条条往下读取之外，它可能会被跳转、中断等改变。

 

SP寄存器：

栈在内存中的地址为$0100-$01FF,共256字节，刚好能被8位的SP所表示。栈的生长方式是传统的从高地址往低地址生长。它不具备stack overflow的检测，所以sp从$00溢出后会重新变为$FF.

 

P寄存器(在有些文档中也被称为状态寄存器-status register)：

P中每一位相当于一个标志位，这些标志位会根据cpu指令而相应地设置或清除。

 

-Carry Flag(C)：进位标志

当出现了上溢出(overflow)和下溢出(underflow)时，C置为1.

上溢出的例子：例如两个16位数相加，会先计算低8位。比如这里两个数的低8位之和为256，超过了255，由于在最高位发生了进位，C置为1.于是在计算高8位时，可以把C中的1作为低8位进上来的位。也就是说，通过C，可以进行超过8位的运算，计算能力被扩展了。

也可以通过SEC指令(置为1)和CLC指令（清除为0）两条指令去改写C。SEC - Set C; CLC - Clear C.

 

-Zero Flag(Z)：零标志

如果前一条指令产生的结果为0，比如128-128就会使得它置1，而128-127就不会。

 

-Interrupt Disable(I):中断禁止标志

此标志可以让系统忽略IRQs发起的中断。可以通过SEI指令去开启这个屏蔽（置为1），或者CLI去关闭屏蔽（清除为0）。

 

-Decimal Mode(D):

这个标志本来可以使6502进入Decimal Mode(前面介绍过了)，不过由于2A03不支持BCD模式，所以尽管这一位还可以被设置，但是并不会产生任何影响。可以通过SED指令去置1，或者CLD指令去清0.

 

-Break Command(B):

这个标志位的意义在于IRQ中断可以由软件引起（执行BRK指令）也可以由外部硬件通过IRQ线引起。那么如何区分这两种中断类型呢？就是通过这一位。具体来说，当中断是由BRK指令产生时，这一位会被置为1；而如果它是被外部硬件中断时，这一位设为0.

 

-Overflow Flag(V):溢出标志

用于标志两个补码(two’s complement)表示的数在运算时产生了溢出，导致了非法的结果。比如两个正数求和产生了负数，或者两个负数求和产生了正数。一个例子，64+64，和应当为正数128，但是它用二进制表示为10000000,按照补码规则，为-128.当产生这样不正确的结果时，V置为1.

 

-Negative Flag(N):负数标志（在有些文档中被简写为S - for Sign）

bit7(最高位)表示符号位，如果它是1的话（表示当前数是负数），N置为1。

 

根据以上的信息，尽管目前还不知道这些寄存器大概要怎么用，不过利用以上信息，我们已经可以写出如下的代码：

    // 寄存器     Word regPC_; // 其中仅有PC寄存器为双字节，其它的都为单字节      Byte regSP_;     Byte regA_;     Byte regX_;     Byte regY_;     Byte regP_;

其中regP由于有很多位，我们希望最好有辅助函数能具体地设置、清除其中每一个标志位。于是可能就有了这样的代码 

enum StatusRegisterFlagEnum : Byte {     CarryFlag = 0,     ZeroFlag,     IntDisableFlag,     DecimalFlag,     BreakFlag,     emptyFlag,     OverflowFlag,     NegativeFlag };  void setStatusRegister(bool condition, StatusRegisterFlagEnum flag) {    if (condition)    {        regP_ |= 0x1 << flag;    } } void clearStatusRegister(StatusRegisterFlagEnum flag) {    regP_ &= ~(0x1 << flag); } Byte getStatusRegister(StatusRegisterFlagEnum flag) {    return (regP_ >> flag) & 0x1; // 用这个方式比起regP & (0x1 << flag)的一个好处是，这样返回的非0即1。在一些情况下，比如我们要取进位标志(CarryFlag)直接参与运算，就会比较方便，否则我们可能需要!!之类的操作 }

* 如果对这些“位操作”，比如&、|、<<、~之类不熟的话，不妨另外搜点文章看，应该不是太难

* 至于为什么只有set需要有个条件参数，而clear没有，这只是因为需要条件去clear的情况似乎没有，到后面可以看到。当然以现阶段来说，无论加不加这个条件，都是合理的，我们并没有预知功能。

 

 

 

4.中断

NES有三种不同的中断：NMI 、IRQ以及reset。不同的中断发生时，应跳往不同的中断响应函数。这个对应关系存在一个叫中断函数向量表中（有点类似现代OS架构里的IDT），位于内存的$FFFA-$FFFF。Vector table一共6个字节，平均分配给3种中断，每个中断两个字节，刚好可以储存一个16位的函数入口地址。

 

当一个中断发生时，系统执行如下的操作：

1.意识到中断发生

2.完成当前指令

3.保存现场（把PC、 P两个寄存器压栈）

4.设置中断屏蔽标志(I)防止新的中断进来

5.从中断向量表中找到当前中断类型对应的中断处理函数地址，跳往那里

6.执行完中断处理函数

7.通过RTI指令(return from interrupt)返回，并且恢复现场（PC, P出栈）

8.继续向下执行原代码

 

三种中断类型：

- IRQs，可屏蔽中断,可以通过中断屏蔽标志(I)去忽略该类型中断。IRQ可以在两种情况下产生，一是软件通过执行BRK(Break)指令来产生；或者硬件通过IRQ线产生外部中断。IRQ中断的处理函数的地址存于vector table中的$FFFE-$FFFF.

 

- NMI(Non-Maskable Interrupt)，不可屏蔽中断。在PPU中，当V-Blank出现在每一帧的最后时，就会产生NMI中断。这个中断不会被中断屏蔽标志(I)所屏蔽，不过可以被PPU的control register1（$2000）所屏蔽。它的频率为60次/秒(在NTSC上)或者50次/秒（在PAL上）。NMI的中断处理函数的地址存于vector table中的$FFFA-$FFFB。它的处理过程如下：

- Reset，当系统刚启动以及用户按下Reset按钮时触发的中断。Reset的中断处理函数的入口存于vector table中的$FFFC-$FFFD。

 

中断的优先级（从高到低）：Reset -> NMI -> IRQs

从中断发生到中断函数被执行之间被定义为有7个CPU cycle。

大部分中断是通过RTI函数返回。不过也有一些例外，例如SquareSoft的“Final Fantasy 1”（最终幻想1）是通过手动控制栈指针，然后执行RTS指令。

 

 

 

 

 

5.CPU指令

一条指令由指令本身(opcode)以及操作数(oprand)组成。比如“JMP $215F”， 意思是跳往内存的$215F和$2160所指的地方，其中指令中的JMP就是指令本身(opcode)，而$215F就是操作数(opcode).指令本身实际上是由一个字节去表示，比如JMP实际上被定义为0x4C或0x6C，取决于它的寻址模式（下面就会讲到什么是"寻址模式"），而$215F需要双字节去表示，因此加起来这是一条3字节的指令。实际上，指令可以是1-3字节长。单字节的指令其实就是仅包含指令本身，比如针对某个特定的寄存器操作，此时是不需要给出任何额外的地址的。而2个字节即除了1个字节的指令以外，还包含1个字节的操作数，如果你还记得前面内存图里的zero page区域，可能能想到整个zero page覆盖的区域都只要1个字节就行了，那么此时操作数就只需一个字节。

咱们再做个推理，既然指令本身是一个字节，那么理论上就有0xFF共256条。不过实际上6502并没有实现所有的可能的opcode，按照功能分，它的指令一共有56种不同的功能。其中有些指令功能在不同的寻址方式下有不同的版本，这样总共有151条有效的， 即151/256。

 

 

 

 

 

6.寻址模式(Addressing Modes)

 

首先我想来通俗地解释一下什么叫“寻址模式”，如果熟悉这个概念的话可以跳过这一段。显然每条指令都是希望完成一定的操作的，比如把一个数取出来++以后再存回某个地方去。不过这个数究竟从哪里取存在很多种可能，比如可能是从某个特定的寄存器取出（这种情况下不需要给出额外的操作数），或者是由操作数直接给出对应的内存地址，它可能是Zero Page的区域(单字节操作数)又或者是其它区域(双字节操作数)，又或者它根本不是直接给出，而是要把操作数中的地址和某个寄存器里的值求和以后才能得到最终要操作的地址等等。以上的种种情况，就对应了不同的寻址模式。因此，同样一个功能的操作(在这个例子中是取出、++、然后储存)由于可以搭配上不同的寻址模式，就变成了不同的指令。

 

6502共有13种寻址方式。

 

1) Implied        （指令长度：单字节）

这种寻址模式下不需要操作数来给出地址，因为指令本身已经包含了。

例如 TAX - 把寄存器A的指给寄存器X

       CLC - 清除进位标志

 

* 这种情况下，由于对象的可能性比较多，比如可能是某个寄存器或者某个标志位等等，所以我直接在对应的opcode的函数里hard-code的，并没有抽象个单独的函数出来。

 

 

 

2) Accumulator    （指令长度：单字节）

有些指令是直接对寄存器A进行操作的，不需要给出额外的操作数

例如 LSR A - 对A逻辑右移一位

 

* 同样没有抽象，直接在opcode的函数中hard-code的.

 

 

 

3) Immediate        （指令长度：双字节）

在这种方式下，操作数的值是在指令中直接给出的。在汇编语言中，立即数之前会有'#'符号。

例如LDA #$0A,它意味着把0x0A这个值读进A寄存器。在机器语言中，不同的寻址方式有不同的机器语言，所以LDA会因寻址方式不同而被翻译成不同的机器语言。在这种模式下，它是$A9 $0A

 参考代码：

static Word addressMode_Immediate(MemoryManager* memMgr, RegisterManager* regMgr) {     Word srcAddr = regMgr->getRegPC() + 1;     return srcAddr; }

可能要说明一下，我个人在目前的代码中把memory由MemoryManager类进行管理，常用的接口包括memory的read\write等；对应的寄存器由RegisterManager类进行管理，常用的接口比如上面提到的几个寄存器的函数,set、get、clear等等，当然这并不是必须的。我的代码仅只用来说明，实现方式随你喜欢啦。

Byte RegisterManager::getRegA() const { return regA_; }

regMgr->getRegPC()获取的是当前的PC寄存器的值，也就是当前的指令本身所在的地址，而regMgr->getRegPC() + 1指的就是跟在指令后面的操作数了，相当于上面例子里的$0A。这里我们把它的地址返回回去。我对于addressMode_函数簇的模式就是只返回换算后的内存地址本身，而不再进一步调用MemoryManager取出的值后再返回。为什么呢？其实我开始也是设计成取出值再返回的，后来发现一些情况下外头需要获得这个srcAddr进行一些操作，于是就统一只返回地址了。

另外一点要注意的是regMgr->getRegPC()返回的是个双字节的数，因为PC寄存器是双字节的。所以这里要用Word，而不能用Byte。

 

 

 

4) Zero Page    （指令长度：双字节）

前面讲到过，Zero Page模式存在的意义就是为了减少指令的长度，提高效率。比如我们要访问的始终是一个字节就能cover到的区域，那就没必要始终传2字节的地址了。在zero page这种寻址模式下，操作数只有1个字节，这样会限制地址为$0000 - $00ff，也就是所谓的Zero Page区域（请参看上面memory map的图）。

 参考代码：

static Byte addressMode_ZeroPage(MemoryManager* memMgr, RegisterManager* regMgr) {     Byte srcAddr = memMgr->memoryReadByte(regMgr->getRegPC() + 1);     return srcAddr; }

注意，我们这里并不是直接返回regMgr->getRegPC() + 1,如果是这样的话就和立即数寻址(Immediate)完全一样了。这里我们需要的内存地址并不是指令的操作数本身，而是存在它对应的内存地址里。所以要再做一次取值的操作。

Byte MemoryManager::memoryReadByte(Word address) {     return memory_[address]; }

对了，memoryReadByte()的定义暂时只是那么简单而已。

 

 

 

5) Zero Page, X （也叫Indexed Zero Page）     （指令长度：双字节）

与上面的一种方式很相似，区别在于求地址时，会将传入的操作数与当前X寄存器中的值相加。例如LDA $80,而当前X寄存器中的值为$0F，则实际要操作的地址为$8F.

注意：如果求和后的地址超过了$FF，则仅保留后几位。例如LDA $80,当前寄存器X中的值为$FF，$80 + $FF = $17F，则实际操作的地址仅取后面一个字节，即$7F.

参考代码：

static Byte addressMode_ZeroPageX(MemoryManager* memMgr, RegisterManager* regMgr) {     Word srcAddr = memMgr->memoryReadByte(regMgr->getRegPC() + 1);     Byte srcAddrWithOffset = (srcAddr + regMgr->getRegX()) & 0xFF;     return srcAddrWithOffset; }

应该不用太多说明。 & 0xFF就是一个截断的操作。当然这里不做显式截断应该也可以，把unsigned short赋给unsigned char会有隐式截断。

 

 

 

6) Zero Page, Y（也叫Indexed Zero Page）     （指令长度：双字节）

这种模式和Zero Page, 唯一的区别就是这里算加法取的是Y寄存器。

参考代码：

static Byte addressMode_ZeroPageY(MemoryManager* memMgr, RegisterManager* regMgr) {     Word srcAddr = memMgr->memoryReadByte(regMgr->getRegPC() + 1);     Byte srcAddrWithOffset = (srcAddr + regMgr->getRegY()) & 0xFF;     return srcAddrWithOffset; }

只是把X替换成了Y

 

 

 

7) Relative（条件分支语句）          （指令长度：双字节）

这种模式用于分支语句。当条件被满足时，一个单字节的值会被加到PC上。注意，这个单字节的值是有符号的，它的取值范围为[-128, 127]。所以说PC和这个单字节的值相加后，可能比原来大也可能比原来小。

例如: BEQ $A7

A7十进制表示为-39.

这条语句会先查看ZERO标志位，如果ZERO标志位为0，表示不需要做加减，正常继续往下走，那么PC会顺序执行下一条语句；而ZERO如果不为0，比如在这个例子中，PC就会减去十进制的39。

参考代码：

static void addressMode_Relative(MemoryManager* memMgr, RegisterManager* regMgr) {     signed int offset = memMgr->memoryReadByte(regMgr->getRegPC() + 1);     signed int regPC = regMgr->getRegPC();     signed int regPCWithOffset = regPC + offset;     regMgr->setRegPC(static_cast<Word>(regPCWithOffset)); }

注意要用signed，因为是有符号的加减。另外这里并没有返回什么地址，因为所有操作就在这个函数里完成了，调用者没什么额外的事了（到下一篇能看到）

 

 

 

8) Absolute     （指令长度：三字节）

在Absolute中，地址作为操作数是直接给出的。它和zero page的区别在于，它是给出16位完整的地址的。

例如  LDA $31F6 

参考代码：

static Word addressMode_Absolute(MemoryManager* memMgr, RegisterManager* regMgr) {     Word srcAddr = (memMgr->memoryReadByte(regMgr->getRegPC() + 2) << 8) | memMgr->memoryReadByte(regMgr->getRegPC() + 1);     return srcAddr; }

Absolute的指令都是三字节的，这意味着我们需要把PC + 1以及PC + 2中的内容组合成一个16字节的地址来看。由于6502是个小端系统，即高位高字节。因此PC+2应该为高8位，PC+1应该为低8位。把他们组合起来可以用 High8Bits << 8 | Low8Bits的方法。

 

 

 

9) Absolute, X (也叫Indexed Absolute)     （指令长度：三字节）

Absolute, X 与Absolute的关系就像Zero Page, X与Zero Page的关系一样：在求地址的时候，加上当前X寄存器的值。例如当前X为$92,则STA $2000，相当于操作$2092

参考代码：

static Word addressMode_AbsoluteX(MemoryManager* memMgr, RegisterManager* regMgr) {     Word srcAddr = (memMgr->memoryReadByte(regMgr->getRegPC() + 2) << 8) | memMgr->memoryReadByte(regMgr->getRegPC() + 1);     Word srcAddrWithOffset = srcAddr + regMgr->getRegX();return srcAddrWithOffset; }

应该不用太多说明

 

 

 

10) Absolute, Y(也叫Indexed Absolute)    （指令长度：三字节）

把Absolute, X中的X寄存器换成Y寄存器就行了

参考代码：

static Word addressMode_AbsoluteY(MemoryManager* memMgr, RegisterManager* regMgr) {     Word srcAddr = (memMgr->memoryReadByte(regMgr->getRegPC() + 2) << 8) | memMgr->memoryReadByte(regMgr->getRegPC() + 1);     Word srcAddrWithOffset = srcAddr + regMgr->getRegY();      return srcAddrWithOffset; }

把X替换成Y就行

 

 

 

11) Indirect（间接寻址）       （指令长度：三字节）

这种模式只适用于JMP指令。它要跳往的地址存于括号中给出的内存地址中（2字节）。

例如 JMP $215F

假设              内存地址    存储的值

                    $215F         $76

                    $2160         $30

则这条JMP指令跳往$3076处(注意是小端系统，所以$30在前)

static Word addressMode_Indirect(MemoryManager* memMgr, RegisterManager* regMgr) {     Word lowOriAddr  = (memMgr->memoryReadByte(regMgr->getRegPC() + 2) << 8) | (memMgr->memoryReadByte(regMgr->getRegPC() + 1));     Word highOriAddr = lowOriAddr + 1;      Byte lowAddr     = memMgr->memoryReadByte(lowOriAddr);     Byte highAddr;     // Handle 6502's 'JMP ($xxFF)' bug     if ((lowOriAddr & 0xFF) == 0xFF)     {         highAddr = memMgr->memoryReadByte(lowAddr & 0xFF00);     }     else     {         highAddr = memMgr->memoryReadByte(highOriAddr);     }      Word srcAddr = (highAddr << 8) | lowAddr;     return srcAddr; }

代码的大部分应该不用解释，不过你有没有发现其中有个部分很奇怪，和上面的说明不一样。就在多出来的这一段：

    // Handle 6502's 'JMP ($xxFF)' bug     if ((lowOriAddr & 0xFF) == 0xFF)     {         highAddr = memMgr->memoryReadByte(lowAddr & 0xFF00);     }

这是在干啥？实际上这是6502的一个广为人知的bug，可以参见这里（http://gunkies.org/wiki/MOS_Technology_6502）

这个bug具体来说是这样的，把前面的例子修改一下

例如 JMP $21FF

假设              内存地址    存储的值

                    $21FF         $76

                    $2200         $30

这里低字节的地址存储在$0x21FF中，而高字节的地址存储在$0x2200中，本来按照前面的规则，最后取出的地址应当还为$0x3076。不过，6502的bug在于当低地址的结尾为0xabFF（此时高字节自然是0xac00,其中c = b + 1）时，实际并不会从$0x2200的地址里取，而是会从$21FF & $FF00也就是$0x2100里面取！不过按照上面参考资料里的说法，它在实际中的影响很小，尤其在用一些知道这个bug的汇编器时，应该是能自动绕过这个问题(存疑)。

 

 

 

 

12) Indexed Indirect(有些地方叫   (Indirect, X)   )     （指令长度：双字节）

在这种模式下，指令中会包含一个zero-page的地址α，然后α会与X中的值求和后给出一个新的地址β,然后得到β中保存的地址γ,最后把γ中的值读到寄存器A中。这说的像绕口令，看一下下面的例子就明白了。

例如  LDA ($3E, X)

假设              内存地址    存储的值 

                     X寄存器      $05

                     $0043        $15

                     $0044        $24

                     $2415        $6E

这条指令执行的顺序:

(i) $3E和$05求和，得到$0043

(ii) 读取$0043,$0044中的值 = $2415

(iii) 读取$2415地址中的值(即$6E)到寄存器A

注意：

a) 第(i)步中的加法，如果发生溢出，那么值是溢出的部分。例如$FF + $2 = $01

b) 在这种模式中，只能用X寄存器

参考代码：

static Word addressMode_IndirectX(MemoryManager* memMgr, RegisterManager* regMgr) {     Byte oriAddr  = (memMgr->memoryReadByte(regMgr->getRegPC() + 1) + regMgr->getRegX()) & 0xFF;     Byte lowAddr  = memMgr->memoryReadByte(oriAddr);     Byte highAddr = memMgr->memoryReadByte(oriAddr + 1);     Word srcAddr  = (highAddr << 8) | lowAddr;     return srcAddr; }

 

 

 

13) Indirect Indexed(或者叫   (Indirect), Y   )          （指令长度：双字节）

这个和上面很类似，只是这儿先读出内存中的值再和寄存器中的值进行加法。假设zero-page给出的值为α，则取内存中地址为α和α+1中值，设为β。把β与寄存器Y中的值求和，假设和为γ。加载内存中地址为γ的值到寄存器A中。

例如: LDA ($4C), Y

假设              内存地址    存储的值 

                      $004C       $00

                      $004D       $21

                      Y寄存器      $05

                      $2105        $06

则以上的执行顺序为

(i) 取出$4C, $4D中的值 = $2100

(ii) 将$2100 与 Y寄存器的值相加 = $2105

(iii) 将内存中$2105处的值(即6D)读到寄存器A中

注意：仅有Y寄存器可以在这种模式中使用

参考代码：

static Word addressMode_IndirectY(MemoryManager* memMgr, RegisterManager* regMgr) {     Byte oriAddr = memMgr->memoryReadByte(regMgr->getRegPC() + 1);     Word srcAddr = (memMgr->memoryReadByte(oriAddr + 1) << 8) | (memMgr->memoryReadByte(oriAddr));     Word srcAddrWithOffset = srcAddr + regMgr->getRegY();     return srcAddrWithOffset; }

 

总结一下寻址模式，按照指令长度分：

单字节：Implied, Accumulator

双字节：Immediate, ZeroPage, ZeroPageX, ZeroPageY, Relative, IndirectX, IndirectY

三字节：Absolute, AbsoluteX, AbsoluteY, Indirect
