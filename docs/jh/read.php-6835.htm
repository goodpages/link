从1980年代以来，CPU有哪些新变化？这些变化又是如何影响程序员的？本文将会为你解答这些问题。

原文链接：http://danluu.com/new-cpu-features/ 
作者：Dan Luu

有人在Twitter上谈到了自己对CPU的认识：

我记忆中的CPU模型还停留在上世纪80年代：一个能做算术、逻辑、移位和位操作，可以加载，并把信息存储在记忆体中的盒子。我隐约意识到了各种新发展，例如矢量指令（SIMD），新CPU还拥有了虚拟化支持（虽然不知道这在实际使用中意味着什么）。

我错过了哪些很酷的发展呢？有什么是今天的CPU可以做到而去年还做不到的呢？那两年，五年或者十年之前的CPU又如何呢？我最感兴趣的事是，哪些程序员需要自己动手才能充分利用的功能（或者不得不重新设计编程环境）。我想，这不该包括超线程/SMT，但我并不确定。我也对暂时CPU做不到但是未来可以做得到的事感兴趣。

下的所有内容除非另有说明，都是指的x86和Linux。历史总是有重演的趋势，很多对于x86是新的事物，对于超级计算机、大型机和工作站来说已经是老生常谈了。

现状

杂记

一方面，芯片有更广泛的寄存器，可寻址更多的内存。在上世纪80年代，你可能已经使用过8位的CPU，但现在你的机器几乎肯定有一个64位CPU。因为我个人人认为你熟悉编程一个64位的机器，我不打算谈论这个太多。除了能够提供更多的地址空间，64位模式（对于32位和64位操作通过x867浮点避免伪随机地获得80位的精度）提供了更多的寄存器和更一致的浮点结果。自80年代初已经被引入x86的其他你非常有可能用到的功能还包括分页/虚拟内存，pipelining和浮点运算。

秘密

我也将避免讨论跟现在不相关的（比如A20M）事，也避免谈论只有当你写驱动程序，BIOS代码，做安全审查，或其他不寻常的低层次的东西（如APIC/x2APIC，SMM或NX位）才能影响到你的事。

内存/缓存 （Memory / Caches）

在剩下的所有话题中，最有可能真正影响到每天编程工作的是内存是如何工作的。我的第一台电脑是一台286。在那台机器上，一个内存访问可能需要几个时钟周期。几年前，我用的是奔腾4系列，内存访问需要花费超过400时钟周期。处理器比内存的发展速度快得多。对于相对较慢内存问题的解决方法是增加了缓存，如果访问模式是可被预测的，它提供了对常用数据更快的访问速度，并预取，预加载数据到缓存。

几个周期与400多个周期比较起来听起来确实很糟;这是完全超过100倍的慢。但是，如果我写了一个对一个很大的有着64位（8字节）值的块上读取并操作的循环，CPU聪明到能在我需要之前就预取正确的数据，这让我在3Ghz的处理器上以约22 GB/s 的速度来处理。在3Ghz运行的一个在一周期可以消耗8个字节的计算只能得到24GB / s的速度，所以获得22GB / s的速度并没有那么糟。我们通过去主记忆体只是丢了8％的性能而不是丢了100倍。

通过使用一阶近似，使用小过你CPU混存的可预测内存访问模式和数据块操作会在现代缓存里实现最大的优势。如果你想尽最大可能的性能好， 这个文件是一个很好的起点。消化了那个100页的PDF文件后，你会想熟悉一下你要优化系统的微架构和内存子系统，以及学习使用像一些类似于likwid的软件来分析和测验你的应用程序。

TLBs

在芯片里也有小的缓存来处理各种各样的事情。除非你真的要全力扑到微优化，你并不需要知道解码指令缓存和其他有趣的小缓存。最大的例外是的TLB们，它们是虚拟内存查找缓存（通过x86上4级页表结构完成）。即使页表是在L1数据缓存，那也将是每次每个查询有4次，或16个周期来完成一次完整的虚拟地址的查询。这对于所有需要被用户模式内存访问的事情是完全不能接受的，从而有了小而快的虚拟地址查找的缓存。

因为第一级TLB缓存必须要快，它被严重地限制了尺寸（一个现代芯片上或许64个项）。如果您使用的4K页面，这限制了您可以在不发生TLB丢失的情况下能够找到的内存数量。x86还支持2MB和1GB的页面；有些应用程序会通过使用较大的页面大小受益匪浅。如果你有一个长时间运行的使用大量内存的应用程序这很值得关注研究一下。

乱序执行/序列化 （Out of Order Execution / Serialization）

对于到目前的几十年间，x86芯片已经可以能够带有思考的执行和再排序执行（以避免因为一个停滞的资源而被阻塞）。这有时会导致很奇怪的表现 。但x86非常严格的要求一个单一的CPU，或者外部可见的状态，像寄存器和记忆体，如果每件事都在按照顺序执行都必须及时的更新。本实施的使用确保了对于任何一对有依赖性的指令，这些指令相对于彼此都是按照正确的顺序来执行。 
这些限制使得事情看起来像按顺序执行意味着，在大多数情况下，你可以忽略OoO执行的存在，除非你试图竭力维持有可能达到的最佳性能。主要的例外情况是你不仅要要确保事情不仅在外部看起来像是按顺序执行，实际上在内部也要真的按顺序。

一个你可能关心的例子是，如果你试图用rdtsc测量出一系列指令的执行时间，rdtsc读出隐藏的内部计数器并将结果置于edx和eax这些外部可见的寄存器。

假设我们做这样的事：

foo rdtsc bar mov %eax, [%ebx] baz

其中，foo，bar 和baz不去碰 eax，edx 或 [%ebx]跟着 rdtsc 的 mov 会把eax的值写入内存中的一些位置，而且因为eax是外部可见的寄存器，CPU将保证直 rdtsc执行后mov才会执行，这会让一切看起来按顺序发生。

然而，因为rdtsc，foo或bar之间没有明显的依赖关系 ，rdtsc可能在foo之前，在foo和bar之间 ，或在bar之后执行都有可能。甚至只要baz不以任何方式影响移动指令也可能存在baz在rdtsc之前执行的情况。有些情况下这会很好，但如果rdtsc被用来衡量foo的执行时间那就不妙了。

为了精确地安排rdtsc和其他指令的顺序，我们需要一个指令来串行化所有的执行。如何准确的做到细节在特尔公司提供的这份本件中 。

内存/并发 （Memory / Concurrency）

上面提到的排序限制意味着相同的位置的加载和存储彼此间不能被重新排序，除此以外，x86加载和存储有一些其他的限制。特别是，对于一个单一的CPU，存储从来不会跟其他存储同时被记录，并且而不管是否是在相同的位置，存储不会与之前的负载一起被记录。

然而，负载可以与更早的存储一起被记录。例如，如果你写

mov 1, [%esp] mov [%ebx], %eax

它被执行起来就像你写

mov [%ebx], %eax mov 1, [%esp]

但反之则不然 - 如果你写了后者，它永远不能像你前面写那样被执行。

你可能通过插入串行化指令迫使第一实例像写起来一样来执行。但是这需要CPU序列化所有的指令。但是这会非常缓慢，因为它有效地迫使CPU要等到所有指令完成串行化后才能执行任何操作。如果你只关心加载/存储顺序，另外还有一个 mfence 指令他会只去序列化加载和存储。

我不打算讨论其他的memory fence，lfence和sfence， 但你可以在这里阅读更多关于他们的内容 。 
我们已经看了单核排序，其中加载和存储大多是有序的，除此以外还有多核排序。上述限制同样适用；如果core 0在观察core1，就可以看到所有的单核规则适用于core1的加载和存储。然而如果core 0和core 1相互作用，不能保证它们的相互作用也是有序的。

例如，core 0和core 1通过设置为0的 eax 和 edx 开始，core 0执行:

mov 1, [_foo] mov [_foo], %eax mov [_bar], %edx

而core 1执行

mov 1, [_bar] mov [_bar], %eax mov [_foo], %edx

对于这两个核来说， eax 必须是 1 因为第一指令和第二指令之间的内芯依赖性。然而， eax 有可能在两个核里都是 0 ，因为核0的第三行可能在核0还没有在核1看到任何东西时执行，反之亦然。

这涵盖了memory barriers，他序列化一个核心内的存储器访问。由于存储被要求在整个内核有一致的顺序，他们可以，他们也影响交叉的核心并发，但要正确的解释清楚这些非常困难。Linus对于使用memory barriers而不是使用locking有这样一段话 ：

不用locking的真正代价最终是不可避免的错误。通过使用memory barriers自以为聪明的做事几乎总是等待发生的错误的前奏。在所有可以发生在十多种不同架构并且有着不同的内存排序的情况下，缺失一个小小的barrier真的很难让你理清楚。…事实上，任何时候任何人编了一个新的锁定机制，他们总是会把它弄错。不要做。

而事实证明，在现代的x86处理器上，使用locking来实现并发通常比使用memory barriers便宜 ，所以让我们来看看锁。

如果我们设置_foo为0，并有两个线程执行 incl (_foo) 10000次，一个单指令同一位置递增20000次，最终保证不会超过20000，但它（理论上）可以低至2 。如果为什么理论最小值为2，而不是10000还不是那么明显，搞清楚这一点是一个很好的练习。如果你觉得显而易见，我的奖励你的练习是，究竟是否任何合理的CPU设计都可能得到这个结果，还是说一些无聊规范规定永远不会发生？在这个帖子里没有足够的信息来回答这个奖励问题，但我相信我已经涉及到了足够的信息。

我们可以用一段简单的代码试验:

#include <stdlib.h> #include <thread>  #define NUM_ITERS 10000 #define NUM_THREADS 2  int counter = 0; int *p_counter = &counter;  void asm_inc() {   int *p_counter = &counter;   for (int i = 0; i < NUM_ITERS; ++i) {     __asm__("incl (%0) \n\t" : : "r" (p_counter));   } }  int main () {   std::thread t[NUM_THREADS];   for (int i = 0; i < NUM_THREADS; ++i) {     t[i] = std::thread(asm_inc);   }   for (int i = 0; i < NUM_THREADS; ++i) {     t[i].join();   }   printf("Counter value: %i\n", counter);   return 0; }

尽管incl是个单独的指令，但不能保证原子性。在内部， incl是后面跟一个add后再跟一个存储的负载。在cpu0里的一个增加有可能偷偷的溜进cpu1里面的负载和存储之间执行，反之亦然。

英特尔对此的解决方案是少量的指令可以加lock前缀，以保证它们的原子性。如果我们把上面代码的incl改成lock incl，输出始终是20000。 
这就是我们怎样把单指令做出原子性。为了使序列atomic，我们可以使用 xchg 或 cmpxchg, 它们始终被锁定为比较和交换的基元。我不会去详细描他是如何工作的，但如果你好奇可以看这篇David Dalrymple的文章。

为了使存储器的交流原子性，lock相对于彼此在global是有序的，而且加载和存储对于锁不会被重新排序相。

所有这些讨论的是硬件的并发是什么和怎样工作的。尽管x86对于什么需要重新排序有限制，编译器确不必具有那些相同的局限性。在C或C++，你需要插入相应的基p以确保编译器不会再排序任何东西。正如Linus在这里指出的 ，如果你有这样的代码：

local_cpu_lock = 1; // .. 做些重要的事 .. local_cpu_lock = 0;

编译器不知道local_cpu_lock = 0不能被放在重要的中间部分。 Compiler barriers与CPU memory barriers不同。由于在x86内存模型是比较严格的，一些编译器的屏障在硬件层面是选择不作为并告诉编译器不要重新排序。如果您使用的是高于microcode，汇编，C或C++语言，编译器很可能没有任何类型的注释为你做这些。

内存/移植 （Memory / Porting）

如果你要把代码移植到其他架构，重要注意的是x86也许有着今天你能遇到的任何架构里最强的内存模式。但如果你不仔细思考只是写能用的代码，把它移植到有较弱担保的架构（PPC，ARM，或Alpha）的架构代码，你将几乎肯定得到报错。

考虑Linus对这个例子的评论：

CPU1         CPU2 ----         ---- if (x == 1)  z = y;   y = 5;     mb();              x = 1;

…如果我读了Alpha架构内存排序保证正确，那么至少在理论上，你真的可以得到Z = 5

mb是memory barrier（内存屏障）。本文不会细讲，但如果你想知道为什么有人会建立这样一个允许这种疯狂行为发生的规范，想一想成产成本上升打垮DEC之前，他们的芯片是快到可以在相同的基准下通过仿真运行跑出比x86在实际中更快的行业标准 。对于为什么大多数RISC-Y架构做出了当时他所做的决定请参见关于Alpha架构背后动机的论文。

顺便说一句，这是我很怀疑Mill架构的主要原因。暂且不论关于他们是否能达到他们号称的性能，仅仅在技术上出色其本身也不是一个商业模式。

内存/非临时存储/写结合存储器 （Memory / Non-Temporal Stores / Write-Combine Memory）

上一节所述的限制适用于可缓存（即“回写（write-back）”或WB）存储器。他曾经在一个时间也是新的。在此之前，只有不可缓存（UC）内存。

其中一个关于UC内存有趣的事情是，所有加载和存储都被设计希望能出来去bus上。这在没有缓存或者几乎没有板载缓存的处理器是完全合理的。这么做的一个结果是，即有机会获得存储的设备可以依赖所有访问UC内存区域创建的单独的总线事务，以便（因为有些设备将使用存储器读或写作为触发做某事的扳机）。在1982年这个工作很伟大，但如果不管任何最新你的视频卡只是想吃下最新的任何更新这就不那么伟大了。如果多次写发生在同一个UC的位置（或同一个词的不同字节），对于每次写CPU需要发出独立的总线交易，即使显卡并不真正关心看不看的到这些干预结果。

该解决方案，这是创建一个名为写结合（WC）内存类型。WC最终是一致性UC。写最终会把它放进存储器，但是它们可以在内部进行缓冲。WC内存相比UC也有较弱的顺序保证。

在大多数情况下，除非你直接与设备交谈，否则你不需要处理这个。一个例外是“非暂时”的加载和存储操作。这些使得特定的负载和存储像他们是WC内存，即使他们在存储区域的地址被标记为WB。

如果你不希望用什么东西污染缓存，这是非常有用的。如果你正在做一些流计算并且你知道只会用一段特定的数据,一次这是非常有用的。

内存/NUMA

非一致内存访问（NUMA），其中内存的延迟和带宽是最不同的处理器不同，是常见我们几乎不再谈论NUMA或CCNUMA，因为它们是如此普遍，被认为是默认的。

这里要求的是共享内存的线程应该在同一个socket上，内存映射I/O重线程应该确保它在最接近的I/O设备的socket对话。

我绝大部分的避免解释事情背后的故事，因为这将使这个帖子至少多十倍的长度。但是我会给一个极大的简化了的为什么我们有NUMA系统，部分原因是它是一个自包含的东西，相比是什么更容易解释多久和部分展示为什么。

曾几何时，只有内存。然后CPU相对于内存速度太快以致于人们想增加一个缓存。缓存与后备存储器（内存）不一致是一个坏消息，因此缓存必须保持它坚持着什么的信息，所以它才知道是否以及何时它需要向后备存储写东西。

这不是太糟糕，但一旦你获得了2个有自己缓存的核，它变得更加有点复杂。为了保持作为无缓存的情况下相同的编程模型，缓存必须相互之间以及与后备存储器是一致的。由于现有的加载/存储指令在他们的API里没有什么允许他们说“对不起!这个加载因为别的cpu在使用你想用的地址而失败了” ，最简单的方式是让每个CPU每次要加载或存储东西的时候发一个信息到总线上。我们已经有了这个两个CPU都可以连接的内存总线，所以我们只要要求另一个CPU在他们数据在其缓存有修改的时候做出数据回复（并失去相应的缓存行）。

这工作起来很好。在大多数情况下，每个CPU只涉及其他CPU不关心的数据，所以有一些浪费的总线流量。但是，这不是太糟糕，因为一旦CPU拿出一条消息说“你好！我要占有这个地址并修改数据”，可以假定在其他的CPU要求前它完全拥有该地址，虽然这将可能不会发生。而且为了替代在一个内存地址上做事情我们可以在一个缓存线上操作，比方说，64个字节。因此，总的多余字节开销是相当低的。

对于4核CPU他依然可以工作，虽然字节浪费相比有点差。但是这件事情，其中每个CPU对其他每一个CPU的响应失败比例远远超出4个CPU的和，既因为总线被饱和，也因为缓存将得到饱和（缓存的物理尺寸/成本是以同时的读和写数量 O(n^2) ，并且速度与大小负相关）。

这个问题“简单”的解决方法是有一个单独的集中目录记录所有的信息，而不是做N路的对等广播。反正因为现在我们正在一个芯片上包2-16个内核，每个芯片(socket)对每个核的缓存状态有个单一目录跟踪是很自然的事。

这不仅解决了每个芯片的问题，而且我们需要通过某种方式让芯片相互交谈。不幸的是，当我们扩展这些系统即使对于小型系统总线速度也快到真的很难驱动一个信号远到连接一堆芯片和都在一条总线上的记忆体。最简单的解决办法就是让每个插座都拥有一个存储器区域，所以每一个socket并不需要被连接到的存储器每一个部分。因为它很明确哪个目录拥有特定的一段内存，这也避免了目录需要一个更高级别的目录的复杂性。

这样做的缺点是，如果你占用一个socket并且想要一些被别的socket拥有的memory，你会有显著的性能损失。为简单起见，大多数“小”（<128核）系统使用环形总线，因此性能损失的不仅仅是通过一系列跳转达到memory付出的直接延迟/带宽处罚，他也用光了有限的资源（环状总线）和减慢了其他socekt的访问速度。

理论上来讲，OS透明得处理这些，但往往低效 。

SIMD

基本上所有现代的x86 CPU都支持SSE，128位宽的向量寄存器和指令。因为要完成多次相同的操作很常见，英特尔增加了指令，可以让你像为2个64位块一样对128位数据块操作，或者4个32位的块，8个16位块等。ARM用不同的名字（NEON）支持同样的事情，而且支持的指令也很相似。

但是，如果你不手写汇编语言，编译器经常会产生非优化的代码 ，特别是对SIMD代码，所以如果你很关心尽可能的得到最佳性能，你就要看看反汇编并检查你编译器的优化错误。

GPU/GPGPU

相比其他部分我不是很够资格来谈论这些。幸运的是，Cliff Burdick自告奋勇地写了下面这节:

2005年之前，图形处理单元（GPU）被限制在一个只允许非常有限硬件控制量的API。由于库变得更加灵活，程序员开始使用处理器处理更常用的任务，如线性代数例程。GPU的并行架构可以通过发射数百并发线程在大量的矩阵块中工作。然而，代码必须使用传统的图形API，并仍被限制于可以控制多少硬件。Nvidia和ATI注意到了这点并发布了可以使显卡界外的人更熟悉的API来获得更多的硬件访问的框架。该库得到了普及，今天的GPU同CPU一起被广泛用于高性能计算（HPC）。

相比于处理器，GPU硬件主要有几个差别，概述如下：

处理器

在顶层，一个GPU处理器包含一个或多个数据流多重处理器（SMs）。现代GPU的每个流的多重理器通常包含超过100个浮点单元，或在GPU的世界通常被称为核。每个核心通常主频在800MHz左右，虽然像CPU一样，具有更高的时钟频率但较少内核的处理器也存在。GPU的处理器缺乏自己同行CPU的许多特色，包括更大的缓存和分支预测。在核的不同层，SMs，和整体处理器之间，通讯变得越来越慢。出于这个原因，在GPU上表现良好的问题通常是高度平行的，但有一些数据能够在小数目的线程间共用。我们将在下面的内存部分解释为什么。

内存（Memory）

现代GPU内存被分为3类：全局内存，共享内存和寄存器。全局存储器是GDDR通常GPU盒子上广告宣称约为2-12GB大小，并具有通过300-400GB /秒的速度。全局存储器在处理器上的所有SMS所有线程都能被访问，并且也是内存卡上最慢的类型。共享内存，正如其名所指，是同一个SM中的所有线程之间共享内存。它通常至少是全局储蓄器两倍的速度，但对不同SM的线程之间是不被允许进行访问的。寄存器很像在CPU上的寄存器，他们是GPU上访问数据最快的方式，但它们只在每个本地线程，数据对于其他正在运行的不同线程是不可见的。共享内存和全局内存对他们如何能够被访问都有很严格的规定，对不遵守这些规则的行为有严重性能下降的处罚。为了达到上述吞吐量，内存访问必须在同线程组间线程之间完整的合并。类似于CPU读入一个单一的缓存行，如果对齐合适的话，GPU对于单一的访问可以有缓存行可以服务一个组里的所有线程。然而，最坏的状况是一组里所有线程访问不同的缓存行，每个线程都要求一个独立的记忆体读。这通常意味着缓存行中的数据不被线程使用，并且存储器的可用吞吐量下降。类似的规则同样适用于共享内存，有一些例外，我们将不在这里涵盖。

线程模型 （Threading Model）

GPU线程在一个单指令多线程(SIMT)方式下运行，并且每个线程以组的形式在硬件中以预定义大小（通常32）运行。这最后一部分有很多的影响;该组中的每个线程必须同一时间在同一指令下工作。如果任何一组中的线程的需要从他人那里获得代码的发散路径（例如一个if语句）的代码，所有不参与该分支的线程会到该分支结束才能开始。作为一个简单的例子：

if (threadId < 5) {    // Do something } // Do More

在上面的代码中，这个分支会导致我们的32个线程中的27组暂停执行，直到分支结束。你可以想象，如果多组线程运行这段代码，整体性能会因大部分的内核处于闲置状态将受到很大打击。只有当线程整组被锁定才能使硬件允许交换另外一组的核来运行。

接口（Interfaces）

现代GPU必须有一个CPU同CPU和GPU内存之间进行数据复制的发送和接收，并启动GPU并且编码。在最高吞吐量的情况下，一个有着16个通道的PCIe 3.0总线可达到约13-14GB / s的速度。这可能听起来很高，但相对于存在GPU本身的内存速度，他们慢了一个数量级。事实上，图形处理器变得更强大以致于PCIe总线日益成为一个瓶颈。为了看到任何GPU超过CPU的性能优势，GPU的必须装有大量的工作，以使GPU需要运行的工作的时间远远的高于数据发送与接收的时间。

较新的GPU具备一些功能可以动态的在GPU代码里分配工作而不需要再回到CPU推出的GPU代码中动态的工作，而无需返回到CPU，单目前他的应用相当有局限性。

GPU结论

由于CPU和GPU之间主要的架构差异，很难想象任何一个完全取代另一个。事实上，GPU很好的补充了CPU的并行工作，使CPU可以在GPU运行时独立完成其他任务。AMD公司正在试图通过他们的“非均相体系结构”（HSA）合并这两种技术，但用现有的CPU代码，并决定如何将处理器的CPU和GPU部分分割开来将是一个很大的挑战，不仅仅对于处理器来说，对于编译器也是。

SMT/超线程 （Hyper-threading）

超线程对于程序员来说大部分是透明的。一个典型的在单核上启用SMT的增速是25％左右。对于整体吞吐量来说是好的，但它意味着每个线程可能只能获得其原有性能的60％。对于您非常关心单线程性能的应用程序，你可能最好禁用SMT。虽然这在很大程度上取决于工作量，而且对于任何其他的变化，你应该在你的具体工作负载运行一些基准测试，看看有什么效果最好。

所有这些复杂性添加到芯片（和软件）的一个副作用是性能比曾经预期的要少了很多;对特定硬件基准测试的重要性相对应的有所回升。

人们常常用“计算机语言基准游戏”作为证据来说一种语言比另一种速度更快。我试着自己重现的结果，用我的移动Haswell（相对于在结果中使用的服务器Kentsfield），我得到的结果可以达到高达2倍的不同（相对速度）。即使在同一台机器上运行同一个基准，Nanthan Kurz 最近向我指出一个例子 gcc -O3 比 gcc –O2 慢25％改变对C ++程序的链接顺序可导致15％的性能变化 。评测基准的选定是个难题。

分行 （Branches）

传统观念认为使用分支是昂贵的，并且应该尽一切（大多数）的可能避免。在Haswell上，分支的错误预测代价是14个时钟周期。分支错误预测率取决于工作量。在一些不同的东西上使用 perf stat （bzip2，top，mysqld，regenerating my blog），我得到了在0.5％和4％之间的分支错误预测率。如果我们假设一个正确的预测的分支费用是1个周期，这个平均成本在.995 * 1 + .005 * 14 = 1.065 cycles to .96 * 1 + .04 * 14 = 1.52 cycles之间。这不是很糟糕。

从约1995年来这实际上夸大了代价，由于英特尔加入条件移动指令，使您可以在无需一个分支的情况下有条件地移动数据。该指令曾被Linus批判的令人难忘的 ，这给了它一个不好的名声，但是相比分支，使用cmos更有显著的加速这是相当普遍的 
额外分支成本的一个现实中的例子是使用整数溢出检查。当使用bzip2来压缩一个特定的文件，那会增加约30％的指令数量（所有的增量从额外分支指令得来），这导致在1％的性能损失 。

不可预知的分支是不好的，但大部分的分支是可以预见的。忽略分支的费用直到你的分析器告诉你有一个热点在如今是非常合理的。CPUs在过去十年中执行优化不好代码方面变好了很多，而且编译器在优化代码方面也变得更好，这使得优化分支变成了不良的使用时间，除非你试图在一些代码中挤出绝对最佳表现。

如果事实证明这就是你所需要做的，你最好还是使用档案导引优化而不是试图手动去搞这个东西。

如果你真的必须用手动做到这一点，有些编译器指令你可以用来表示一个特定分支是否有可能被占用与否。现代CPU忽略了分支提示说明，但它们可以帮助编译器更好得布局代码。

对齐 （Alignment）

经验告诉我们应该拉长struct，并确数据对齐。但在Haswell的芯片上，几乎任何你能想到的任何不跨页的单线程事情的误配准为零。有些情况下它是有用的，但在一般情况下，这是另一种无关紧要的优化因为CPU已经变得在执行不优良代码时好了很多。它无好处的增加了内存占用的足迹也是有一点害处。

而且， 不要把事情页面对齐或以其他方式排列到大的界限，否则会破坏缓存性能 。

自修改代码 （Self-modifying code）

这是另外一个目前已经不怎么有意义的优化了。使用自修改代码以减少代码量或增加性能曾经有意义，但由于现代的缓存倾向于拆分他们的L1指令和数据缓存，在一个芯片的L1缓存之间修改运行的代码需要昂贵的通信。

事务内存支持正如它听起来这样：事务的硬件支持。通过三个新的指令xbegin、xend和xabort。

xbegin开始一个新的事务。一个冲突（或xabort）使处理器（包括内存）的架构状态回滚到在xbegin的状态之前.如果您使用的是通过库或语言支持的事务内存，这对你来说应该透明的。如果你正在植入库支持，你就必须弄清楚如何将有有限的硬件缓冲区大小限制的硬件支持转换成抽象的事务。

快速I/O（Fast I/O）

对于存储和网络来说,I/O带宽正在不断上升，I/O延迟正在下降。问题是，I/O通常是通过系统调用完成。正如我们所看到的，系统调用的相对额外费用一直在往上走。对于存储和网络，答案是转移到用户模式的I / O堆栈 （把一切放在内核模式下也将起作用，但这个更难推销）。在存储方面，这仍然是一个很怪的事情来研究，但HPC和HFT在网络中已经这样做了一段时间了。关于一段时间，我的意思不是几个月。下面是从2005年的论文，讲的是我要讨论的网络的东西，以及一些我不打算讨论的东西（DCA）。

最终慢慢进入了不是超级计算机的世界。MS已经给有虚拟化RDMA有InfiniBand网络的Azure做了一年多广告，CloudFlare已经谈到使用Solarflare的网卡来获得相同的能力等，最终，我们将看到有很快板载Ethernet的SoC，除非这仅限于Xeon的设备，它将下到所有设备。ARM器件之间的竞争很有可能至少会引起一个ARM装置制造商把快速以太网放在其商品SoC，这可能会迫使英特尔也出手。

那个RDMA位是很有意义的;它可以让你完全绕过CPU和让NIC响应远程请求。几个月前，我曾学习研究了斯坦福/Coursera挖掘海量数据集类的课。在第一堂课期间，他们提供了一种“典型的”用架顶式1Gb交换机组成的数据中心的例子。如果你正在做内核的TCP通过非RDMA网卡，那处理“海量”的数据不是不合理的，因为你可能让整个核心试图以1Gb / s的速度通过Linux的的TCP协议栈推。但随着Azure，MS谈到在一台机器获得的40Gb;这是在一台机器上获得40倍的你希望在整个机架上获得的宽带。他们还提到负2微秒的延迟，这比你可以在内核TCP中得到的大小低了很多。这不完全是一个新的想法。2011年的这份论文预测了到目前为止网络方面发生的每件事，还有一些东西，还有一些完全不相关的事。

这篇MS的讲座讨论了对于网络存储你可以怎样利用这类带宽和延迟。一个具体的不需要通过点击一个链接的例子是亚马逊的EBS。它可以让你在任何您的AWS节点上使用一个任意大小的“弹性”磁盘。由于旋转金属盘的寻道在TCP内核比RPC具有更高的延迟性 ，你可以非常透明的得到无限的存储。例如，假设你在你的网络得到100us（0.1ms）的延迟 ，你的磁盘寻道时间为8ms。这使得远程磁盘访问变成8.1ms，而不是8毫秒，这没有那么多的浪费开销。那个与SSD的工作起来就没那么好，虽然，因为你可以在固态硬盘得到20us（.02ms）的延迟 。但RDMA延迟足够低到对固态盘来说透明的类EBS状层是可能的。

所以那是网络I/O。如果/当下一代比闪存更快的存储技术得到开始得到运用，在硬盘方面其性能优点可能更大。性能的变量如此之大以至于英特尔正在增加新的指令以跟上下一代低延迟存储技术 。根据你问的是谁，这些东西已经有十几二十年的时间了;这是比网络的东西更加不确定。但即使有Flash，人们依然炫耀可以得到分解成单个微秒级延迟的设备，这在本质上是一个很大的改进。

晶体管规模化一个有趣的副作用是我们可以把很多晶体管包进一个芯片上，但他们产生如此多的热量，如果你不希望你的芯片融化，普通晶体管大多数时间不能开关。

这样做的结果把包括大量时间不使用的专用硬件变得更有意义，。一方面，这意味着我们得到各种类的专用指令PCMP和ADX指令。但是，这也意味着，我们正把整个曾经不集成在芯片上的设备与芯片集成。这包括诸如GPU和（用于移动设备）无线电。 
与硬件加速的趋势相结合，这也意味着企业设计自己的芯片，或者至少自己芯片的部分变得更有意义。通过收购PA Semi公司，苹果公司已经走出了很远。首先，加入少量定制的加速器给停滞不前的标准的ARM架构，然后添加自定义加速器给他们自己定制的架构。由于正确的定制硬件和基准和系统设计深思熟虑的结合，iPhone 4比我的旗舰级Android手机反应还稍快，这个旗舰机比iPhone4新了很多年，并且具有更快的处理器以及更大的内存。

亚马逊挑选了很好的旧Calxeda的团队的一块，并雇用了一个足够大小的硬件设计团队。Facebook也已经挑选了ARM SoC的专家，并与高通公司在某些事情 - 或 - 其他方面合作。Linus也有纪录在案的说，“我们将在各个方面看到更多的专用硬件” 。等等。

结论

x86芯片已经有了很多新的功能和非常成功的小工具。在大多数情况下，要利用这些优势你不需要知道他们是什么。作为一级近似，使您的代码可预测并记住内存位置很好。真正的底层的东西通常是由库或驱动程序隐藏了起来，编译器将尝试照顾其余部分。例外是，如果你真的写底层代码，这种情况下世界上已经变得更加混乱，或者如果你想在你的代码里获得绝对的最佳表现， 这种情况下世界已经变得更加怪异。

此外，事情会在未来发生。但大多数的预测是错误的，所以谁又知道呢？
