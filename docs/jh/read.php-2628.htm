近距离观察GPU part1

GPUs: A Closer Look原文
前言：

新一代的GPU马上要出来了，很快各种评测以及架构分析的文章就会大量涌现。当然很多都是忽悠人的。翻译这篇文章的目的是希望能为对GPU真正感兴趣的人提供一些真正严谨的介绍，少受误导。本文翻译自Kayvon Fatahalian和Mike Houston所写的GPUs：a closer look一文，版权属于原作者，不希望被转载。 原文非常长，我一部分一部分贴上来，大家也好充分消化，文中括号里的内容多是我的一些注解，方便大家理解，不保证正确性。Ok，一起进入奇妙的GPU世界吧。

现代GPU可以被视作一种通用处理器，它和STI的cell，sun的UltraSPARCT2以及来自Intel和AMD的多核X86处理器一道成为多核并行计算架构中令人瞩目的一部分。与传统的CPU不同，对GPU而言，维持大量并行操作的高吞吐量远比以较低的延时（low latency）完成单项任务重要。GPU融合了大量固定功能（fixed function)单元以及可编程单元，有大量的ALU单元和很高的峰值浮点运算能力。尽管图形处理具有天生的并行性质，但是要把通常的绘制算法映射到GPU架构上仍然具有相当的挑战性。高性能GPU的关键是设计出好的硬件架构以及有与之相匹配的软件接口（software interface)来释放出硬件庞大的计算能力。GPU设计的目标是获得高性能的同时保持一个简单有力的编程接口（我认为这个接口是比OpenGL或者D3D API更低一级的接口，由硬件暴露出来），以此方便程序员编程。这个编程接口没有显示并行（explicit parallelism）和异步执行（asynchronous execution）的特征，并且在每代GPU之间以及不同厂商之间保持兼容。


当CPU平台转向以吞吐量为中心（throughput-oriented）的设计原则时，面临着并行编程的复杂性难题（想想现在有多少程序能真正利用好双核，四核的CPU）。因此理解GPU架构成功的关键因素不但有助于开发GPU程序，也对设计其他领域的新架构和编程系统有帮助。在这篇文章中，我们将拨开现代GPU上的层层迷雾，来探索为什么交互式绘制（interactive rendering）是如此有挑战性以及如何设计GPU来克服这些挑战。


图形流水线（The Graphics Pipeline）


图形系统的功能是生成表示虚拟场景某个视图的图像（其实可以把图形系统理解为照相机，拿照相机拍摄一幅照片就等效于图形系统渲染一幅图像）。虚拟场景的外观由几何体（primitives），方向，物体表面的材质以及光源所在的位置和光源性质决定（稍微有点OpenGL和D3D编程经验的人或者做电脑动画的人都不会对这些概念陌生）。图形系统试图在获得最高性能和保持简洁有力的编程接口间获得一个很好的平衡（想想Cell，性能很强，但是编程也很困难）。

当前用于实时绘制的API，例如OpenGL和D3D，通过把渲染过程抽象为图形处理流水线（Graphics ProcessingPipeline）来获得上面提到的良好平衡。在图形处理流水线中，处理的对象包括：顶点（vertices），几何图元（primitives），片元（fragment），像素（pixel）。图1展示了一个简化的七级图形流水线。数据以流的形式（streamentities）在流水线中流动。这条简化的流水线中，包含了固定功能的单元（图中黄褐色部分）和可编程单元（棕褐色部分）。图2展示了流水线中关键模块的功能。

VG（Vertex Generation，顶点生成）实时图形API用一系列简单的几何图元（点，线，三角形）来表示物体表面。每一个图元由一些列顶点定义。开始绘制时，应用程序首先把顶点描述符送到VG单元（可以把顶点描述符想象成一份清单，告诉VG需要哪些顶点），VG通过顶点描述符从显存中取得所需的顶点并构建顶点数据流以便做后续处理。在实际程序中，每个顶点除了包含（x，y，z）三维坐标外还有额外的应用程序自己定义的顶点属性，例如顶点颜色，法向等。

VP（Vertex Processing，顶点处理）VP是可编程的，即应用程序可以指定让VP做什么处理（可以把VP当作vertex shader）。VP的输入和输出是一一对应的，即一个顶点被VP处理后仍然是一个顶点（VP不会增加或者减少顶点，注意与Geometry Shader的区别）。各顶点间的处理是相互独立的。VP最重要的功能是计算把三维顶点映射到二维屏幕上的投影矩阵。

PP（Primitive Processing，几何处理）同VP类似，PP也是可编程的（可以把PP当作Geometry Shader）。PP的输入和输出不再是一一对应的。一个图元被PP处理后可以生成0个或者多个图元，即PP可以增加或者减少几何图元。因此PP的输出是一系列新的几何图元。PP对各图元处理也是相互独立的。

FG（Fragment Generation，片元生成）FG对每一个几何图元在屏幕空间中（Screen Space）进行采样（这一过程被称之为光栅化）。每一个采样点显然就是一个片元。片元记录了该采样点在屏幕空间中的位置，与视点之间的距离以及通过插值获得的顶点属性。

FP（Fragment Processing，片元处理）FP模拟了光线和物体表面的交互作用，以此决定每个片元的颜色以及透明程度等属性。为了获得更加真实的效果，FP还大量使用了纹理滤波。FP也是可编程单元（可以把FP当作pixel shader）。

PO（Pixel Operations）PO用每个片元的屏幕坐标信息来计算该片元对最终生成图像上的像素的影响程度。PO计算每个采样点离视点的距离，并丢弃哪些被遮挡住的片元。当来自多个几何图元的片元影响同一个像素时（最常见的情况是半透明物体的相互遮挡），很多绘制技术都依赖于PO来根据几何图元的位置按序的更新像素。所有的图形API都要确保PO的正确，同时PO阶段也是唯一由流水线本身来决定以何种顺序处理片元的阶段。

Shader编程 （Shader Programming）


对于图形流水线中的可编程单元（VP，PP，FP），它们的行为是由应用程序定义的，即编写相应的shader函数（shader functions）。图形程序能够用高级绘制语言（例如nvidia的Cg，OpenGL的GLSL，微软的HLSL）来编写vertex shader，geometry shader，pixel shader。Shader程序被离线编译为字节码（byte code），然后在运行时由驱动程序将其转化为依赖于GPU（GPU-Specific）的二进制程序。

绘制语言支持复杂的数据类型和丰富的控制流指令，但是绘制语言并没有显示并行的特性。因此，一个shader程序类似于C语言中的函数，它顺序的计算每个输入流的输出。每一个输入流对应同样的shader程序（例如不能让一些像素执行一个pixel shader，另外一些像素执行另外的pixel shader），shader程序在各个输入流上的执行是相互无关的。

为了方便，除了输入流和输出流外，shader程序可以读取（但是不能修改）一些大的，全局共享的缓存（buffers）。在流水线启动前，这些缓存就被初始化，包含了一些跟shader相关的参数以及应用程序提供的纹理。

特征和挑战（Characteristics and Challenges）

图形流水线具有以下一些关键特征：

并行处理（Opportunities for parallel processing）。图形渲染具有任务并行（比如在图形流水线中，shader单元在执行shader程序，同时Z单元在通过Z值来剔除被遮挡的图元，这两个任务是并行执行的）和数据并行（比如shader单元的SIMD）的特征，这使得可以通过并行处理来提升吞吐率。尽管图形处理具有高度并行性，但是PO阶段的处理要求数据有序。这一限制使得图形流水线的设计复杂化。例如在PO阶段中，对大部分的片元是可以并行处理的，但那些对应同一个像素的片元是不能并行处理的。

固定功能单元负责处理难于并行化的工作（Fixed-function stages encapsulatedifficult-to-parallize work）。虽然每一个shader程序的执行是按序的（即shader程序里的每一条指令是应当按序执行的），但是同一个shader程序可以并行的执行在不同的输入数据流上（想象pixel shader程序被同时执行在不同的像素上），这体现了可编程单元最简单的并行性。与之相反，图形流水线中的非可编程单元负责解决那些难于并行的任务，例如PO阶段对于有序的要求以及PG阶段对顶点的重组）。把这些非数据并行任务（non-data-parall）同可并行任务隔离开，并交给固定功能单元处理，使得shader的编程模型得以简化，同时也使得GPU的的可编程单元变得高度专用化（highly specilaized），即非常适合处理数据级并行的任务。另外，这种分离设计使得图形计算中那些难于处理的任务被封装到了高度优化的固定功能单元中。

流水线负载的极度不均衡（Extreme variations in pipeline load）。虽然图形流水线的处理阶段和数据流向是固定的，但是各个阶段对于计算能力和带宽的需求变化非常大，这取决于shader程序和渲染场景的复杂度（例如 仙剑三和crysis对于计算能力和带宽的需求完全不在一个数量级上）。例如，一个很大的几何图元可能产生很多的片元（fragment），对像素处理提出了很高要求；相反很小的三角形只产生很少的片元，因此对顶点处理能力要求更高。应用程序中的shader程序的复杂程度也有天壤之别，有的shader程序包含十几条指令，有些则包含数百条指令。由于以上原因，图形流水线的不同阶段中，有些是带宽密集型需求（bandwidth intensive），有些是计算密集型需求（computeintensive），并且需求总是在不断变化，因此要随时都能把图形流水线的不同需求有效的映射到GPU资源上是非常具有挑战性的，因为这要求对GPU的资源能够根据当前工作负载来动态分配。

数据存取方式的可预测性和不可预测性兼而有之（Mixture of predicatable andunpredictable data access）。在图形流水线中，各阶段的数据流是固定的，可预测的（例如总是顶点处理->几何处理->像素处理）。这种可预测性使得我们有机会采用激进的预取策略（aggressive prefetch）来存取数据流中的数据和GPU的资源。与之相反，shader程序对于缓存和纹理数据的存取则是动态变化的，使得预取变得困难。因为这两种数据存取方式对于维持GPU的高吞吐率都很关键，shader的编程模型对于这两种存取方式做了显示区分，使得可以设计高度专用和优化的硬件来分别解决这两类存取方式。

指令流的共享（Opportunities for instruction stream shadring）。在GPU中，每一个线程都保存自己的shader指令。在实际中，由于局部性原理，相邻的线程常常按照相同的动态控制流（dynamic control-flow）执行指令。因此shader指令可以在不同的线程GCD享。虽然GPU也必须要处理例外情况，但是在多线程间共享shader指令流是优化GPU处理核心和流水线调度算法的关键。（这一段不太好翻译，大家看不懂就跟帖问吧，我会尽力解答。）

总结：第一部分主要介绍了一些关于图形流水线和GPU架构的一些基础知识，给大家一个感性认识。下一部分将深入到GPU架构的内部，探讨当今GPU架构是怎样设计的，以及为什么会这样设计。
