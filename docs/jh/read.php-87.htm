  80.    non terminals that have an integer value are listed.  An object  
  81.    value means that it can be any type, it isn’t set to a specific  
  82.    type.  So it could be an Integer or a String or whatever. */   
  83. non terminal Object     expr_list, expr_part;   
  84. non terminal Integer    expr, factor, term;  
  85.   
  86. /* ————-Precedence and Associatively of Terminals Section———– */   
  87. /*  
  88.   Precedence of non terminals could be defined here.  If you do define  
  89.   precedence here you won’t need to worry about precedence in the  
  90.   Grammar Section, i.e. that TIMES should have a higher precedence  
  91.   than PLUS.  
  92.   The precedence defined here would look something like this where the  
  93.   lower line always will have higher precedence than the line before it.  
  94.   precedence left PLUS, MINUS;  
  95.   precedence left TIMES, DIVIDE;  
  96. */  
  97.   
  98. /* —————————-Grammar Section——————– */   
  99. /* The grammar for our parser.  
  100.    expr_list ::=   expr_list expr_part  
  101.                  | expr_part  
  102.    expr_part ::=   expr SEMI  
  103.    expr      ::=   expr PLUS factor  
  104.                  | expr MINUS factor  
  105.                  | factor  
  106.    factor    ::=   factor TIMES term  
  107.                  | factor DIVIDE term  
  108.                  | term  
  109.    term     ::=    LPAREN expr RPAREN  
  110.                  | NUMBER  
  111.                  | ID      
  112. */   
  113. /* ‘expr_list’ is the start of our grammar.  It can lead to another  
  114.    ‘expr_list’ followed by an ‘expr_part’ or it can just lead to an  
  115.    ‘expr_part’.  The lhs of the non terminals ‘expr_list’ and  
  116.    ‘expr_part’ that are in the rhs side of the production below need  
  117.    to be found.  Then the rhs sides of those non terminals need to be  
  118.    followed in a similar manner, i.e. if there are any non terminals  
  119.    in the rhs of those productions then the productions with those non  
  120.    terminals need to be found and those rhs’s followed.  This process  
  121.    keeps continuing until only terminals are found in the rhs of a  
  122.    production.  Then we can work our way back up the grammar bringing  
  123.    any values that might have been assigned from a terminal. */   
  124.    expr_list ::= expr_list expr_part   
  125.                  |   
  126.                  expr_part;   
  127. /* ‘expr_part’ is an ‘expr’ followed by the terminal ‘SEMI’.  The ‘:e’  
  128.    after the non terminal ‘expr’ is a label an is used to access the  
  129.    value of ‘expr’ which will be an integer.  The action for the  
  130.    production lies between {: and :}.  This action will print out the  
  131.    line " = + e" where e is the value of ‘expr’.  Before the action  
  132.    takes places we need to go deeper into the grammar since ‘expr’ is  
  133.    a non terminal.  Whenever a non terminal is encountered on the rhs  
  134.    of a production we need to find the rhs of that non terminal until  
  135.    there are no more non terminals in the rhs.  So when we finish  
  136.    going through the grammar and don’t encounter any more non  
  137.    terminals in the rhs productions will return until we get back to  
  138.    ‘expr’ and at that point ‘expr’ will contain an integer value. */   
  139.    expr_part ::= expr:e   
  140.                  {: System.out.println(" = " + e); :}   
  141.                  SEMI   
  142.                  ;   
  143. /* ‘expr’ can lead to ‘expr PLUS factor’, ‘expr MINUS factor’, or  
  144.    ‘factor’.  The ‘TIMES’ and ‘DIVIDE’ productions are not at this  
  145.    level.  They are at a lower level in the grammar which in affect  
  146.    makes them have higher precedence.  Actions for the rhs of the non  
  147.    terminal ‘expr’ return a value to ‘expr’.  This value that is  
  148.    created is an integer and gets stored in ‘RESULT’ in the action.  
  149.    RESULT is the label that is assigned automatically to the rhs, in  
  150.    this case ‘expr’.  If the rhs is just ‘factor’ then ‘f’ refers to  
  151.    the non terminal ‘factor’.  The value of ‘f’ is retrieved with the  
  152.    function ‘intValue()’ and will be stored in ‘RESULT’.  In the other  
  153.    two cases ‘f’ and ‘e’ refers to the non terminals ‘factor’ and  
  154.    ‘expr’ respectively with a terminal between them, either ‘PLUS’ or  
  155.    ‘MINUS’.  The value of each is retrieved with the same function  
  156.    ‘intValue’.  The values will be added or subtracted and then the  
  157.    new integer will be stored in ‘RESULT’.*/   
  158.    expr      ::= expr:e PLUS factor:f   
  159.                  {: RESULT = new Integer(e.intValue() + f.intValue()); :}   
  160.                  |   
  161.                  expr:e MINUS factor:f   
  162.                  {: RESULT = new Integer(e.intValue() – f.intValue()); :}   
  163.                  |   
  164.                  factor:f   
  165.                  {: RESULT = new Integer(f.intValue()); :}   
  166.                  ;   
  167. /* ‘factor’ can lead to ‘factor TIMES term’, ‘factor DIVIDE term’, or  
  168.    ‘term’.  Since the productions for TIMES and DIVIDE are lower in  
  169.    the grammar than ‘PLUS’ and ‘MINUS’ they will have higher  
  170.    precedence.  The same sort of actions take place in the rhs of  
  171.    ‘factor’ as in ‘expr’.  The only difference is the operations that  
  172.    takes place on the values retrieved with ‘intValue()’, ‘TIMES’ and  
  173.    ‘DIVIDE’ here instead of ‘PLUS’ and ‘MINUS’.  */   
  174.    factor    ::= factor:f TIMES term:t   
  175.                  {: RESULT = new Integer(f.intValue() * t.intValue()); :}   
  176.                  |   
  177.                  factor:f DIVIDE term:t   
  178.                  {: RESULT = new Integer(f.intValue() / t.intValue()); :}   
  179.                  |   
  180.                  term:t   
  181.                  {: RESULT = new Integer(t.intValue()); :}   
  182.                  ;   
  183. /* ‘term’ can lead to ‘LPAREN expr RPAREN’, ‘NUMBER’, or ‘ID’.  The  
  184.    first production has the non terminal ‘expr’ in it so the  
  185.    production with its lhs side needs to be found and followed.  The  
  186.    next rhs has no non terminals.  So the grammar ends here and can go  
  187.    back up.  When it goes back up it will bring the value that was  
  188.    retrieved when the scanner encounter the token ‘NUMBER’.  ‘RESULT’  
  189.    is assigned ‘n’, which refers to ‘NUMBER’, as the action for this  
  190.    production.  The same action occurs for ‘ID’, except the ‘i’ is  
  191.    used to refer to ‘ID’.  ‘ID’ is also the only thing on the rhs of  
  192.    the production.  And since ‘ID’ is a terminal the grammar will end  
  193.    here and go back up. */   
  194.    term      ::= LPAREN expr:e RPAREN   
  195.                  {: RESULT = e; :}   
  196.                  |   
  197.                  NUMBER:n   
  198.                  {: RESULT = n; :}   
  199.                  |   
  200.                  ID:i   
  201.                  {: RESULT = i; :}   
  202.                  ;
