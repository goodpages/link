OpenGL(3)——Win32/OpenGL 程序框架

1.概述

OpenGL(1)中建立了一个MFC单文档应用程序，并在其中加入OpenGL初始化和简单绘图的相关代码，实现了在MFC快速开发环境中的OpenGL编程，效率高。但是这样做只知其然，不知其所以然，对开发者清楚了解OpenGL中各相关代码在整个程序中的位置、作用和调用时机等，都没有太大的好处！开发者只能按照文中所讲的一一修改，然而到底为什么修改，在哪里修改，都不够深入。MFC开发的好处是让开发者将精力放在更重要的代码实现上面，如OpenGL绘图、绘图控制等环节上，但是作为学习OpenGL的开发者，还是应该将OpengGL的代码从MFC的结构中独立出来，并在单纯的Win32程序中实现其基本设置和简单的绘图工作，这样，才能真正发挥OpenGL独立于开发平台的开放性图形开发库的特色，才能真正掌握这门技术。

2.Win32 SDK程序框架

Win32软件开发包（SDK）是在Windows 32位机平台下的软件开发包，包含了各类API函数及其相关套件。Win32程序总是依托于窗口，采用事件驱动方式，基于消息机制。这种消息处理的机制可用下图表示：


图：Windows 的消息处理机制

图中，1是操作系统感知应用程序窗口上发生的事件，并将事件包装成消息（也就是用具体数据填充一个MSG结构体，这个结构体的对象就是一个消息），例如用户在程序窗口上点下了鼠标左键，那么操作系统感知这一事件，并将它包装成一个消息叫WM_LBUTTONDOWN消息；2是操作系统将消息投递到该应用程序的消息队列中，消息队列是应用程序开始执行后，系统为程序创建的。应用程序通过一个循环语句不断地从队列中取出操作系统投递过来的消息（当然也有不进队列的消息）；3是应用程序调用GetMessage( )函数从消息队列中取出一条消息，应用程序可以对消息进行一些预处理，如放弃消息的响应、调用TranslateMessage( )产生新的消息等；4是应用程序调用DispatchMessage( )函数，将消息回传给操作系统；5是操作系统通过窗口类注册时传入的窗口过程函数名（也就是窗口过程函数的指针）调用应用程序对应的窗口过程函数，对消息进行具体的处理，开发者可以事先定义好窗口过程函数，在其内定义好对哪些消息进行处理，怎样处理等等。

MFC是一套面向对象的函数库，以类的方式提供给开发者，它将Windows API函数封装到类中。所以MFC的底层还是Win32程序。每个Win32应用程序都要完成固定的任务，MFC也不例外，比如要定义一个窗口类，注册窗口类，要创建窗口，进行消息循环、窗口过程定义等。但是MFC的工程代码里找不到这些明显的定义代码，因为MFC底层框架类中封装了这些部分，只有当程序编译链接时，由链接器将那些基本的函数链接到MFC程序中，同时，MFC基于消息映射机制，可能跟Win32程序的直接消息处理方法上有所不同，但是本质是一样的。所以，从更高级的MFC代码中拆出OpenGL相关的代码，将其还原到Win32框架中来，应该是不难的，关键是找准各修改部位的调用原理。

3.对应的修改

OpenGL(1)中，要对MFC中的以下函数进行了修改：

PreCreateWindow( )

OnCreate( )

OnSize( )

OnEraseBkgnd( )

OnInitialUpdate( )

OnDraw( )

OnDestroy( )

其中，PreCreateWindow( )和OnInitialUpdate( )函数是两个虚函数，前者是CWnd类的成员函数，它在窗口被完全创建之前调用，其主要功能是为开发人员提供一个修改窗口类属性的接口，后者是CView类的一个虚函数，下面会详细讲到。

其余的5个函数是消息响应函数。分别对消息WM_CREATE、WM_SIZE、WM_ERASEBKGND、WM_PAINT和WM_DESTROY进行消息响

应。那么我们可以在Win32程序框架中的窗口过程函数中，用switch-case语句定义这些消息的处理方法。但是在定义之前，弄清楚这些消息分别是在什么时候被触发的很重要。

4.WM_CREATE消息的响应

WM_CREATE消息的触发与响应原理如下图所示：


图：WM_CREATE消息的响应

在WinMain( )函数中，首先定义一个窗口类，然后将定义好的窗口类注册到操作系统中。然后由CreateWindow( )函数按照定义好的窗口类各属性创建一个窗口实体。在CreateWindow( )函数执行的过程中，操作系统引发了一个WM_CREATE消息，如果成功创建窗口后返回一个窗口句柄，但是，这个时候的窗口实体还不可见，要用到ShowWindow( )函数才能最终让窗口实体显示出来。对应于窗口过程函数中的OnCreate( )函数专门用来响应WM_CREATE消息，它的设置就是在窗口已被创建并拥有了一个句柄，但是窗口实体又没有显示的这个时机，可以利用返回的窗口句柄，初始化一些代码。所以我们可以在Win32程序的窗口过程中定义WM_CREATE消息的处理，主要可以完成一些获取窗口DC和像素格式设置的工作。

至于WM_SIZE消息，它是在窗口尺寸发生变化的时候触发的，仍然在窗口过程函数中定义，比如将RC设置为当前可用的状态，然后设置OpenGL投影变换等。WM_ERASEBKGND消息是在窗口区域中的内容变得无效时，如Invalidate(TRUE)函数，WM_PAINT消息，窗口尺寸改变都会先触发一个WM_ERASEBKND消息将窗口客户区中的内容全部擦除掉（非客户区的擦除与重绘则由操作系统自行搞定）。我们的代码中不希望在尺寸发生改变的时候先擦除窗口中的内容，我们希望直接重绘覆盖先前的内容。所以，我们可以在这个消息的处理中直接返回TRUE，从而重载默认的擦除功能。WM_DESTROY消息也很简单，可以在响应代码中加入删除RC的处理。

5.WM_PAINT消息

事实上，MFC中的OnDraw( )函数并不是WM_PAINT消息的直接响应者。其直接调用者是OnPaint( )函数，只是在OnPaint( )函数中，它调用了OnDraw( )函数。那么为什么搞得这么复杂呢？实际上，出现OnDraw( )是为了实现各种不同的设备上的绘图一致性。

实际上，出现OnDraw( )是为了实现各种不同的设备上的绘图一致性。


首先，WM_PAINT消息是为了绘制屏幕而出现的，因此，在OnPaint( )中重载函数，我们能做的是存取屏幕DC(DC有四种，详细情况可以参看《关于DC》)，然后进行绘制，常见的代码是：


void MyWnd::OnPaint()
{
       CPaintDC dc(this);
      //绘制代码
}

这里的CPaintDC的构造函数会自动调用BeginPaint，获得一个屏幕DC，并附加在dc对象上。当dc对象析构时，系统自动调用EndPaint并使invalidated rectangle变成validated状态，从而结束绘制。(注意，重复创建CPaintDC实例会失败也因为如此)。

如果我们在OnPaint（）中绘制，那么在打印机上绘制我们就需要再写一个OnPrint（）函数，重新绘制。这样，程序设计者就需要维护两套代码。为了简化操作，MFC框架把大部分绘制操作都放在OnDraw（）中统一管理要显示的内容，OnPaint（）和OnPrint（）只构造相应的DC，然后分别调用OnDraw（）。也就是说，OnDraw（）适用于所有的设备（前提是调用它的函数能取得相应设备的DC），而OnPaint（）只适用于屏幕绘制。

还有一点需要说明的就是，在响应WM_PAINT消息的时候，如果要获取DC，不能使用GetDC( )（释放时用ReleaseDC( )函数），而只能使用BeginPaint( )（释放时用EndPaint( )函数）。因为之所以有WM_PAINT消息，就是因为窗口需要重绘，也就是说窗口中的内容当前已无效。BeginPaint( )和EndPaint( )完成后可以删除消息队列中的WM_PAINT消息，并使无效区域有效。GetDC( )和ReleaseDC( )并不删除也不能使无效区域有效，因此当程序跳出WM_PAINT时，无效区域仍然存在。系统就回不断发送WM_PAINT消息，于是程序不断处理WM_PAINT消息。相当于BeginPaint、EndPaint会告诉GDI内部，这个窗口需要重画的地方已经重画了，这样WM_PAINT处理完返回给系统后，系统不会再重发WM_PAINT，而GetDC没有告诉系统这个窗口需要重画的地方已经画过，在你把程序返回给系统后，系统一直以为通知你的重画命令你还没有乖乖的执行或者执行出错，所以在消息空闲时，它还会不断地发WM_PAINT催促你画，导致程序卡死。所以，在响应WM_PAINT消息时，要清楚使用哪个函数来取得DC。

对于我们的Win32程序，我们只用OnPaint( )在屏幕上绘制，所以取消调用一个OnDraw( )函数，获取DC，并添加绘制代码。与MFC的调用结果应该是一样的。

5.两个虚函数重载

MFC中还有两个函数要修改，一个是CWnd::PreCreateWindow( )函数中修改窗口类的某些属性。CWnd::PreCreateWindow( )是一个虚函数，它是由MFC设计出来以提供开发者修改窗口某些属性的一个接口。默认情况下，MFC定义了一套窗口类，但是这个窗口类不一定满足开发者的需要，所以要能够修改这个窗口类，或者窗口。在MFC中，一个窗口创建出来之前，可以调用CWnd::PreCreateWindow( )函数来修改CREATETRUCT结构体。那么在Win32程序中，我们可以直接在CreateWindow( )的参数DWORD dwStyle来设定一个窗口的风格，加入WS_CLIPCHILDREN | WS_CLIPSIBLINGS，这样，也能完成MFC中的修改任何。这两个窗口风格位标识符是将影响窗口绘制的其它窗口的重叠区域剪切掉。

另一个是CView::OnInitialUpdate( )函数。CView类的派生类可以重载CView类中的虚函数OnInitialUpdate( )函数。在派生类中，于OnInitialUpdate( )之前，可以添加自己的初始化代码，进行初始化工作。然而，OnInitialUpdate( )的内部实现是什么？为什么不在OnCreate( )，或者视类的构造函数中进行初始化工作呢？它们之间肯定有所不同。


首先，CView::OnInitialUpdate( )是在视类第一次与文档类关联时，被框架类所调用的，而这个时候，视类还没有真正显示出来。这个函数是一个虚函数，其内的实现，就是简单地调用OnUpdate( )函数（其中，对OnUpdate( )函数的三个参数分别赋予NULL, 0, NULL）。


那么，CView::OnUpdate( )又是怎样实现的呢？在视类文档中的文档被修改后，框架调用该函数。默认的实现方法就是使整个客户区无效，于是触发WM_PAINT消息引起窗口客户区的重绘。


这样看来，实际上如果没有添加自己的代码，‍CView类的派生类中OnInitialUpdate( )函数中其实就是使窗口无效，即调用Invalidate(true)函数。那么‍CView::OnUpdate( )函数又是具体在什么时候被程序调用的呢？CView::OnUpdate( )是视图窗口完全建立后，也就是CreateWindow( )函数返回之后，第一个被框架调用的函数，随后框架会第一次调用OnPaint( )中的OnDraw( )函数。因此OnInitialUpdate是设置绘图的相关初始化参数最合适的地方。


那么，为什么不在视类的构造函数，或者在OnCreate( )消息响应中处理这些相关初始化呢？首先在时间上，构造函数生成视类对象，却没有产生出窗口，当然还没有窗口句柄，没有窗口像素格式，OnCreate( )返回后，有了窗口，但是窗口还不可见。然后才到视类的OnInitialUpdate( )函数，一般也就在这里对视图的显示做初始化工作了。所以我们可以在这里进行诸如显示列表等的创建。
