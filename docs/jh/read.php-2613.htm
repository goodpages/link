OpenGL(4)——OpenGL类的设计

1.概述

自然地，如果将OpenGL相关代码从Win32程序中作某种代码的分离，并将其封装成类，那么对图形程序开发会产生一定的好处。进一步，将这些类写入DLL文件中，则整个程序更具有便捷性和隐蔽性。但是从另一方面来讲，清爽的代码会将复杂性遮蔽起来，对学习OpenGL的深层次原理可能不太有利，所以事先得对OpenGL有一个比较清楚的把握，但这不是本文所要讲解的内容。

OpenGL的部分初始化代码是必须的，也是烦人的。这些代码一旦设置清楚，后面基本都不必改动，所以C++类应该可以将这些代码封装起来以使代码更便于重用。

这里，我们将封装一个C++类，它叫作“CGL”。它内部的一些OpenGL代码，特别是独立于开发平台的那些代码（比如说绘图的函数glBegin( )等等），是直接由前面讲的小程序中剪切并粘贴出来的，所以基本没有多大的改动，我们把这些代码叫作纯粹OpenGL代码；但是那些与操作系统有关的OpenGL设置代码（比如说SetPixelformat( )等等）应该被封装起来，我们把这些代码叫作Win32/OpenGL代码。所以CGL类要封装比如说设置RC、DC、像素格式等各种程序都要用到的设置代码，对于那些纯粹OpenGL代码，则再用其它的类封装。这种设计模式有点像是OpenGL提供的辅助库，辅助库中提供了各种初始设置的函数，直接调用就可以创建出窗口，绘制显示等等，但是那些纯粹OpenGL代码命令都并没有封装起来。同时，CGL类设计可以运用到MFC程序中来，所以也要考虑到可能会借用MFC内的某些基类，比如说CView类。

2.CGL的设计思路

设计一个类，有两个设计思路，一个是函数封装模式，一个是结构封装模式。它们不是相互排斥的，因为结构封装中也会大量运用到函数封装。

2.1函数封装模式

在函数封装中，纯粹OpenGL命令是CGL类的成员函数。这些函数通过一个CGL类对象调用。比如下面的例子：

glMatrixMode(GL_PROJECTION) ;

glLoadIdentity( ) ;

gluPerspective(30.0, gldAspect, 1.0, 10.0) ;

glViewport(0, 0, cx, cy) ;

那么用函数封装后就是：

CGL gl ;

gl.glMatrixMode(GL_PROJECTION) ;

gl.glLoadIdentity( ) ;

gl.gluPerspective(30.0, gldAspect, 1.0, 10.0) ;

gl.glViewport(0, 0, cx, cy) ;

这样做有几个好处，函数封装使OpenGL命令更加地面向对象，通过对象调用函数，编程时不用一个函数一个函数地写；可以在类中重载很多同类的函数，于是可以在编程时减少函数的种类，比如说在OpenGL中用于设置颜色的命令就有5种：glColor3f、glColor3d、glColor3i、glColor4f、glColor4uiv，通过C++函数重载，每次设置颜色的时候，只用一个函数编译器就会根据函数参数自行执行对应的函数；此外这种方法还可以避免数据类型不匹配等的编译器警告……

但是，函数封装也会带来不好的东西，它隐藏了OpenGL的语法，函数封装必然要对原函数进行必要修改，那么你不可能直接从其它程序代码中复制代码到自己的程序中来，当然也不能从自己的程序中代码复制到别人的程序中去，更痛苦的是将OpenGL函数整个封装下来，可能会使自己的头发白掉一半——太费时了。

同时，我们知道，要使用OpenGL命令，必须要有一个当前活动的RC。我们可以使用wglMakeCurrent( )函数让一个RC当前可用。同时在OpenGL(2)中我们已知道，管理RC的方法有一种是从程序的一开始就保持一个RC当前可用，然后在整个程序的生命同期中都维护一个与之关联的DC。那么，假设如果某个程序要渲染两个分离的场景：一个场景在窗口的状态栏；一个场景在客户区。每个场景都要有一个RC，那么我们这样写代码：

CGL glStatusBar ;

CGL glClientArea ;

glStatusBar.Color(1.0, .0, .0 ) ;

glClientArea.Color(.0, 1.0, .0 ) ;

glStatusBar.CallList(STATUS_BAR) ;

glClientArea.CallList(CLIENT_AREA) ;

为了让它们都能运行起来，每个函数（Color和CallList）都要调用wglMakeCurrent( )使一个RC可用，比如说其中一个函数就应该这样写：

void CGL::Color(GLdouble r, GLdouble g, GLdouble b)

{

if (m_hrc != wgGetCurrentContext( ) )

wglMakeCurrent(m_pdc->m_hDC, m_hrc) ;

glColor3d(r, g, b) ;

}

如果这样做的话，无疑会使程序冗杂不堪。怎么办？用结构封装试试。

2.2结构封装模式

结构封装的目的是封装OpenGL程序的结构，隐藏那些Windows的实现细节，也就是‍Win32/OpenGL代码；而纯粹OpenGL代码就不作为CGL的类成员函数了，或者说作为类成员函数中被调用的东西。我们借用一下OpenGL(1)中提到的几个函数，并把它们对应到我们的CGL类中：

class CGL

{

public:

BOOL Create( ) ;

BOOL Init( ) ;

BOOL Resize( ) ;

BOOL Render( ) ;

void Destroy( ) ;

}

显然，CGL中包含的是程序的结构，而不是具体的那个OpenGL函数。这个类告诉我们，CGL应该具有5种方法，应该完成5种功能。如果我们在Win32程序框架中使用这个类，可以在相应的消息响应中调用这些方法；如果在MFC中使用这个类，那么我们可以在MFC的相应消息响应函数中调用CGL类中的成员函数，如下图所示：


图：MFC消息处理函数调用CGL类中的成员函数

在设计中，CGL::Create主要完成RC、DC的创建任务；CGL::Init则用于初始化OpenGL参数并设置显示列表；CGL::Resize完成场景到屏幕的投影；CGL::Render将场景送进屏幕；CGL::Destroy则完成类的最后清理工作。

好，我们以OpenGL(1)中讲到的OnSize函数为例，看一下到底怎么用结构封装处理这个函数：

void CGLEasyView::OnSize(UINT nType, int cx, int cy)
{
CView::OnSize(nType, cx, cy)
if ( (cx <= 0) || (cy <= 0) ) return ;
CClientDC dc(this) ;

wglMakeCurrent(dc.m_hDC, m_hrc);

//****************************************************************************
// 建立三维物体到二维屏幕的投影方法

GLdouble gldAspect = (GLdouble) cx/ (GLdouble) cy;
glMatrixMode(GL_PROJECTION);// OutputGlError("MatrixMode") ;
glLoadIdentity();
gluPerspective(30.0, gldAspect, 1.0, 10.0);
glViewport(0, 0, cx, cy);

wglMakeCurrent(NULL, NULL);
}

这个代码的底下几行函数（从GLdouble gldAspect 开始），是GLEasy程序所特有的，不具有一般性（即其它程序中的代码可能就不这么写了）。我们不想把这些代码放到CGL类中，于是我们要把那些对于某些程序所特有的代码从一般性的OpenGL代码区分开来，这里某些程序所特有的代码也就是前面所提的纯粹的OpenGL代码，而一般性的OpenGL代码也就是前面提到的Win32/OpenGL代码。那么上面的代码，我们要封装的是wglMakeCurrent函数而已，而对于程序特有的代码，我们可以放到CGL的派生类中去，见下图所示：


图：程序特有的代码放入派生类

基类为：

class CGL

{

public:

BOOL Create( ) ;

virtual BOOL Init( ) ;

virtual BOOL Resize(int cx, int cy) ;//基类中主要作判断和置RC为当前可用

virtual BOOL Render( ) ;

void Destroy( ) ;

}

派生类为：

class CScene : public CGL

{

protected:

virtual BOOL Resize(int cx, int cy) ;

virtual BOOL Init( ) ;

virtual BOOL Render( ) ;

}

派生类中的虚函数的实现：

BOOL CScene :: Resize(int cx, int cy)

{

CGL::Resize(cx, cy) ; //调用父类的Resize(cx, cy)函数，完成窗口尺寸判断和wglMakeCurrent

GLdouble gldAspect = (GLdouble) cx / (GLdouble) cy ;

glMatrixMode(GL_PROJECTION);
glLoadIdentity();
gluPerspective(30.0, gldAspect, 1.0, 10.0);
glViewport(0, 0, cx, cy);

//全生命周期保存RC为当前可用，所以不用wglMakeCurrent(NULL, NULL);

}

也许这样很好，在派生类的虚函数中先调用父类的Resize( )完成wglMakeCurrent( )，然后可以放心使用纯粹OpenGL代码。在程序中利用派生类对象使用派生类重载的虚函数完成功能。

这是一种分离特有与一般的方法，但是这样的安排不一定适用于Render( )函数，因为Render( )函数要在使用纯粹OpenGL代码之前，调用Win32/OpenGL代码 wglMakeCurrent( )，在使用完 纯粹OpenGL代码后又调用 Win32/OpenGL代码SwapBuffer( )函数。那么怎样安排这一前一后的 Win32/OpenGL代码呢？当然我们可以再为CGL添加一个成员函数完成SwapBuffe，但是我们希望每个基类的成员函数在型式上统一，不要突然出现一个 SwapBuffe成员函数。我们可以到MFC中取一下经：OpenGL(3)中讲到过，MFC中的CWnd::OnPaint( )函数在调用CView::OnDraw( )函数之前调用了BeginPaint，然后在调用CView::OnDraw( )函数之后调用EndPaint( )。同样的道理，我们可以用一个基类的非虚函数Render( )来调用一个纯虚函数OnRender( )，而在派生类中对OnRender( )进行实现，只完成 纯粹OpenGL代码，开发者可以根据自己的渲染需要任意重载这个函数；而在Render( )函数中完成这个一前一后的‍Win32/OpenGL代码实现。这样以来，Render( )函数整体上就将一个混杂的代码的结构封装起来。下面是一个简化了的CGL类的定义：

class CGL
{
public:

//基类中成员函数在型式上统一。
BOOL Create() ;
BOOL Init() ;
BOOL Resize() ;
BOOL Render() ;
void Destroy() ;

void MakeCurrent() ; //封装wglMakeCurrent( )，专供每一个成员函数内部使用。
protected:
virtual BOOL OnResize() = 0 ;
virtual BOOL OnInit() = 0 ;
virtual BOOL OnRender() = 0 ;
}

派生类中的虚函数完成纯粹OpenGL代码的实现，通过重载可以满足不同程序的不同需要：

class CScene : public CGL
{
protected:
virtual BOOL OnResize() ;
virtual BOOL OnInit() ;
virtual BOOL OnRender();
}

下图说明了CGL类的结构体系：


图：CGL类到其派生‍‍类的执行路径

而CGL::Render( )函数主要把精力放在Win32/OpenGL代码的实现上，如下所示：

BOOL CGL::Render()
{
// 将RC置为当前可用状态
MakeCurrent() ;

// 渲染

OnRender() ;

// 交换缓存
SwapBuffers(m_pdc->m_hDC) ;

return TRUE ;
}

当然，这里CGL有了一个成员函数叫makeCurrent( )，并将wglMakeCurrent封装到了makeCurrent( )函数中，这是明智的。因为CGL总是保证在纯粹OpenGL代码被调用之前，RC是当前可用的状态。像其它的CScene类中的虚函数在调用之前，都要用到这个成员函数使用RC当前可用。下面是MakeCurrent( )函数的实现：

void CGL::MakeCurrent()
{

if (m_hrc != wglGetCurrentContext()) //检查RC要不要改变，如果不用改变则直接使用先前的可前可用的RC
wglMakeCurrent(m_pdc->m_hDC, m_hrc);
}

这里，还有一个问题，就是在基类中定义了一个成员函数，也声明了一个纯虚函数，那么这个基类就是一个抽象基类，抽象基类是不能被实例化的，但是可以定义一个基类指针，通过基类指针调用其内的函数，这是可以的。那么在一个基类的成员函数体内，再调用一个没有被实现的（仅仅只是作了一个声明）的纯虚函数，让抽象基类的派生类来实现这个纯虚函数。这种情况下，在使用类时，一般可以有三种方法：

*** 实例化一个派生类对象，通过对象调用派生类中由基类继承下来的成员函数；

CScene ascene ;

ascene.Create( ) ;

*** 建立一个派生类的指针，将派生类对象地址给指针变量，通过指针变量调用派生类中的函数

CScene ascene ;

CScene* m_pCS = &ascene ;

*** 建立一个基类指针，由动态创建一个派生类对象，由基类指针调用基类中的成员函数

CGL* m_pGL = new CScene ;

一般使用第三种方法，动态创建一个派生类对象。这种方法充分体现了C++的面向对象特征，数据的隐蔽性等。从基类指针的调用来看，它只能调用基类中的成员函数，但是基类中的成员函数中又包含了由派生类来实现的纯虚函数，派生类帮忙定义好了这些纯虚函数，所以虽然是调用了基类的成员函数，但是其功能在整体上是完整无缺的，其中有派生类的功劳。

好，以上就是CGL类在设计时的一些思路和注意事项，依此类推，可以将其它的类成员函数一一写出来！心中只要清楚：哪些代码是 纯粹的OpenGL代码，哪些是 Win32/OpenGL代码，前者由纯虚函数通过开发者自行重载实现，后者则要我们在CGL类成员函数中事先设置好！

3.将CGL类放入DLL(动态链接库)

DLL就像是一个容器，内装有各种函数和类。通常它不能直接运行，也不接收消息，它们是独立的文件，只有当其它模块调用DLL中的函数时，它们才会发挥作用。C++类与DLL一起总是让问题变得比较麻烦，但是MFC扩展DLL（AFXDLL）可以在一定程度上简化从一个DLL导出C++类的过程，它为MFC开发者提供了方便。

将CGL类入到一个DLL中，叫作GLlib.DLL。那么以后各种不同的程序在使用CGL类时，要调用GLlib.DLL。每次我们可以对DLL中的CGL类作一些改动，那么不同的程序可以不用改动地利用CGL类。想想，如果把CGL类直接放到每个需要它的程序中，那么相当于每次都要修改程序，如果放到了DLL中，对于不同的程序，我只用改DLL使之适应即可。后面将详细介绍将一个类加入DLL的方法。

4.使用CGL类

下面是使用CGL类的步骤：

*** 用AppWizard工具编译应用程序框架

*** 加入OpenGL头文件（GL\GL.H和GL\GLU.H）和CGL.H, CScene.h

*** 将OpenGL库文件（OPENGL32.LIB和GLU32.LIB）加入到工程Link中

*** 将CGL库文件（GLlib或者是GLlib-d.DLL）加入工程LINK中

*** 如果在MFC工程中使用这个类，可在CView类中加入一个数据成员，这个成员是上面派生类的一个实例（简单讲一下实例与对象的区别：对象是类的抽象形式，实例是对象的具体化，比如说CString str ; 说明str是CString类的一个对象，也可以说str实例化了CString类；继续 str = "abc" ; 说明abc是类对象str的一个实例，显然一个对象可以有很多实例。）

*** 如果在Win32程序框架中使用这个类，可在全局变量的位置设置一个CGL* m_pGL = new CScene ，然后在所有消息响应中利用这个基类指针调用相应函数。

最后有一点注意就是，在Win32程度框架中，在WinProc，即窗口过程函数中，对WM_PAINT消息的响应中，要获得一个DC句柄，一律只用BeginPaint( )函数来取得，最后用EndPaint( )函数释放DC，而不用GetDC( )。同时，考虑到代码的可读性和完整性，还是统一在每个要使用DC的地方获取DC，使用完后及时释放DC，虽然我们知道获取DC后就算不释放DC，程序结束时也会为我们释放这个DC。
