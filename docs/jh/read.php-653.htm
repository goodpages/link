本文结合C、lua、javascript三种语言着重谈了词法作用域的问题。


初见闭包，是在大二上看小骆驼的时候; 第一次遇到词法作用域，是在大二下看Lua黄皮书的时候，而和调用链结合起来，则是刚刚。事实上直到现在，我才确定我彻底理解了这几个概念。下面的内容百分之九十是摘抄字网络上各种各样的资料，剩下的百分之十是原创。

1.1 闭包(closure)

关于闭包，在线的资源还是很多的，我就不献丑了。

以下摘自维基百科

 在计算机科学中，闭包（Closure）是词法闭包（Lexical Closure）的简称，是引用了自由变量的函数。这个被引用的自由变量将和这个函数一同存在，即使已经离开了创造它的环境也不例外。所以，有另一种说法认为闭包是由函数和与其相关的引用环境组合而成的实体。

闭包的概念出现于60年代，最早实现闭包的程序语言是Scheme。之后，闭包被广泛使用于函数式编程语言如ML语言和LISP。很多命令式程序语言也开始支持闭包。

在一些语言中，在函数中定义另一个函数时，如果内部的函数引用了外部的函数的变量，则可能产生闭包。运行时，一旦外部的 函数被执行，一个闭包就形成了，闭包中包含了内部函数的代码，以及所需外部函数中的变量的引用。其中所引用的变量称作上值(upvalue)。

闭包一词经常和匿名函数混淆。这可能是因为两者经常同时使用，但是它们是不同的概念。


以下引用自IBM developerWorks


闭包并不是什么新奇的概念，它早在高级语言开始发展的年代就产生了。闭包（Closure）是词法闭包（Lexical Closure）的简称。对闭包的具体定义有很多种说法，这些说法大体可以分为两类：

  • 一种说法认为闭包是符合一定条件的函数，比如参考资源中这样定义闭包：闭包是在其词法上下文中引用了自由变量(注 1)的函数。
  • 另一种说法认为闭包是由函数和与其相关的引用环境组合而成的实体。比如参考资源中就有这样的的定义：在实现深约束(注 2)时，需要创建一个能显式表示引用环境的东西，并将它与相关的子程序捆绑在一起，这样捆绑起来的整体被称为闭包。

这两种定义在某种意义上是对立的，一个认为闭包是函数，另一个认为闭包是函数和引用环境组成的整体。虽然有些咬文嚼字，但可以肯定第二种说法更确切。闭包只是在形式和表现上像函数，但实际上不是函数。函数是一些可执行的代码，这些代码在函数被定义后就确定了，不会在执行时发生变化，所以一个函数只有一个实例。闭包在运行时可以有多个实例，不同的引用环境和相同的函数组合可以产生不同的实例。所谓引用环境是指在程序执行中的某个点所有处于活跃状态的约束所组成的集合。其中的约束是指一个变量的名字和其所代表的对象之间的联系。那么为什么要把引用环境与函数组合起来呢？这主要是因为在支持嵌套作用域的语言中，有时不能简单直接地确定函数的引用环境。这样的语言一般具有这样的特性：

  • 函数是一阶值（First-class value），即函数可以作为另一个函数的返回值或参数，还可以作为一个变量的值。
  • 函数可以嵌套定义，即在一个函数内部可以定义另一个函数。

2 词法作用领(lexical scope)

词法作用域，又叫静态作用域(static scope)，这个名词是和所谓动态作用域相对的。

要理解lexical scope，首先要了解scope。scope就是程序语言中查找变量的规则。那么变量又是什么意思呢，我们从最简单的C语言说起：

int x = 7;    // declare global x - x1

int proc(int x)  // declare local x - x2
{
    if (x % 3 == 0) {    // reference  x - x2 right value
        do {
            int y = x * 77;    // reference x - x2 right value
            int x;        // declare x - x3
            x = y;        // reference x - x3 left value
            return x;      // reference x3 as right value
        } while (0);
    } else
        return x;        // reference x2    right value
}

int main(int argc, char **argv)
{
    int y = atoi(argv[x]);    // reference x1    right value
    return proc(y)
}

让我们给程序中的变量x分分类（见注释)
  • 声明 declaration
  • 引用 reference
    • 左值 left value
    • 右值 right value
在C语言中，scope（即作用域，下文同）由一对花括号所限定，scope分为两种，语句scope可以任意嵌套，而函数scope不能嵌套（记得吗，C中函数只能定义在最外层）。假设C编译器支持C89（而非C99），所有的声明都集中在scope（块）的最上部。

因为scope可以嵌套，我们把一串层层嵌套的scope称为作用域链(scope chain)。每次声明就把一个新的符号名与一个变量（内存位置建立关联），而每次引用，如果是作为右值，就是读取这个内存位置的当前值，如果是作为左值，就是写入内存位置。

C语言中在同一个cope内声明并使用的变量又被称为局部变量(local variable)，在所有scope外声明的变量可以在任意的scope内引用（除非被同名的内层变量所遮蔽），还有一种声明在引用语句scope之外的scope中的变量，对所有标示符所指向的变量的引用遵循以下的规则：
  1. 首先在当前scope中查询标示符
  2. 如果不存在则在紧挨着的外层scope中查询，依次类推
  3. 如果在所有scope中都没有找到同名的标示符，则在全局变量中查找。
在C语言的例子中，你可能有点奇怪，这三条规则不能用一条简单的规则来替代吗？只需要把全局变量看作隐含的最外层大scope，那么用层层嵌套的scope chain规则就可以解释清楚了。

的确，对C语言来说，这样是完全正确的，但本文的重点不在C上。因为C没有第一类(first-class的函数，且函数定义不可嵌套，这大大简化了lexical-scope规则。对于支持完整的lexical scope的语言来说，一条规则是不够的。

1.1 作用域规则

  1. 这些语言安全不支持作用域，举个例子，unix工具家族中的dc(desktop calculator)，只能使用a到z这26个字母作为变量名（dc中叫做寄存器）。大多数汇编语言也是如此。
  2. C语言，支持lexical scope，但不支持嵌套函数。在C中，你不可能用同样的标示符访问另一个函数中的局部变量。但可以用指针（内存地址）的方式，这是C的强大之处。
  3. 完整的支持Lexical scope，但不支持first-class函数，换句话说，它们是命令式的。例如pascal，algol这样的经典语言。
  4. 完整支持lexical scope并支持first-class函数。一般来说这样的语言都支持闭包，例如scheme,javascript，Lua等等，在这种语言中所有的scope都对应于匿名函数，所以不存在函数scope和非函数scope。首先查找当前函数、再查找定义当前函数的外层函数、再查找外层的外层函数……如果按照实现方法来分类，还分为两种类
    1. Deep Binding: 较为常见，标示符被绑定到函数被定义的环境中
    2. Shadow binding： 大多数宏系统使用这种规则，标示符被绑定到函数被调用的环境中。
  5. 动态作用域（dynamic scope）：一般也是函数式语言（例如Lisp），假设所有函数都拥有一张key-value表，key是标示符字符串、而value就是内存地址。对变量引用的查询从当前函数开始、再查找调用当前函数的函数、再查找调用当前函数的函数……
全局变量或者叫自由变量不受作用域规则的约束，一般它们都被放在一个全局的表中，由所有函数共享。

比如，下面是一段javascript代码：
var x=0;                         /*  x 是全局变量 */
function f(s){        
        eval(s)
        alert(x)
}
f("var x=1");       /* 这段代码不在任何作用域内，"x"指向全局变量x */

其实这是一种元编程手段，在更高的层次上编程。上面的例子是生成一段字符串并将其当作代码执行，在Lua可以有下面更直接了当的办法
x = 0;         -- global variable

a = function(global_env, key, value)
    global_env[key] = value
end

a(_G,"x", 1)
print(x)              -- 全局变量x现在是1

面向对象

如果一个语言中同时有面向对象和闭包的话，那可真是一件麻烦的事情。学习javascript是很久之前的事了，依稀记得js也是按照self语言的方法来面向对象的（这个术语有争议）。

apply方法把一个对象当作self对象来运行函数，那么在函数内部self变量就是在运行是绑定的，实际上就是个额外参数吗。这样对对象变量（实际上是通过额外参数self进行的）的访问就不需要通过scope规则了，因为self此时是一个局部变量（参数）。
