对象类型
RTL 有五种对象：表达式、整数、宽整数、字符串和向量。其中，最重要的是表达式。RTL 表达式（RTX）类似于 C 的结构，通常用指针来引用它。这种引用它的指针的类型定义名为rtx。
整数相当于C 中的int，用十进制表示。宽整数是HOST_WIDE_INT 类型的一个整数对象，其书写形式也用十进制表示。
字符串为一串字符，在存储器中以C 的ch ar*形式表示且按 C 语法规定书写。 RTL中的字符串决不会为空值。若机器描述中有一空字符串，它在存储器中则表示成一个空指针而不是通常意义上的指向空字符的指针。在某些上下文中，允许用这种空指针表示空字符串。在 RTL 代码中，字符串经常出现在symbol_ref 表达式中，但也出现在某些机器描述的 RTL 表达式中。
一个向量由任意多个指向表达式的指针组成。一个向量中的元素数目是确定的。向量的书写形式为 【exp1 ...expn】，exp 是指向表达式的指针，exp 之间以空格隔开。不生成长度为 0 的向量，而以空指针代替。表达式根据“表达式代码”（又称 RTX 代码）分类。RTX 代码是文件 rtl.def中定义的一个名字，类似于 C 语言中的一个枚举常量。表达式的代码和意义与机器无关，我们可用 GET_CODE (x) 宏从 RTX 中抽取 RTX 代码，用 PUT_CODE (x, newcode) 来改变 RTX 代码。
RTX 代码决定了一个表达式所含的操作数个数及操作数的对象类型。与Lisp 不同的是，在RTL 中不能通过操作数本身而得知操作数是何种对象，而必须从操作数的上下文，即 RTX 代码决定其对象类型。例如：在代码为subreg 的表达式中，它的第一个操作数是一个表达式，第二个操作数是一个整数。在代码为plus的表达式内则有两个均为表达式的操作数。在symbol_ref 表达式内，只有一个被视作为字符串的操作数。
表达式的书写形式为：
（RTX代码 【机器方式】 操作数1 … 操作数 n）
其中，RTX 代码即RTL 表达式代码的名字，机器方式是可选的。各操作数之间用空格分开。
表达式代码名字在机器描述文件中是小写方式，但当出现在 C 代码中时，以大写方式存在。

RTL类型

GNU CC 将所有RTX 代码分为如下几种类型（用单个字符表示）：
? o 表示一个实际对象，如寄存器（reg），存贮位置（mem，symbol_ref），对象的常数和基本转换（ ADDRESSOF ， HIGH ， LO_SUM ） 等。但 subreg 和STRICT_LOW_PART 不属于此类，而属于‘x’类。
? < 表示比较运算，属于这类的代码有 NE、EQ、LE、LT、GE、GT、LEU、LTU、GEU，GTU。
? 1 表示一元算术操作。如 NEG、NOT、ABS，值的扩展（sign/zero）和整型和浮点转换。
? c 表示可交换的二元操作，如PLUS、AND，但不包括 NE 和 EQ，它们属于‘<’类。
? 2 表示不可交换的二元操作，如 MINUS、DIV、ASHIFTRT。
? b 表示位域操作，目前只有ZERO_EXTRACT 或 SIGN_EXTRACT。
? 3 表示其它三个输入参数的操作。目前只支持IF_THEN_ELSE。
? i 表示一个完整的指令，包括INSN、JUMP_INSN 和 CALL_INSN。
? m 表示指令（INSN）内的匹配对象，如 MATCH_DUP，只在机器描述中出现。
? a 表示自动增加的寻址模式，如POST_INC。
? g an rtx code for grouping insns together (e.g, GROUP_PARALLEL)
? x 表示其它所有的RTX 代码。包括用于机器描述的其它代码，所有描述副作用的代码（SET、USE、CLOBBER 等）以及出现在指令链中的非指令代码，如NOTE、BARRIER和CODE_LABEL。对每一种表达式代码，文件 rtl.def 中定义了它包含的操作数个数以及操作数的类别。我们称表示各操作数类别的一串字符串为表达式代码格式，如subreg 的格式是 ‘ei’。

最常使用的格式字符为：
e 代表表达式（实际上为指向表达式的指针）。
i 代表整数。
w 代表宽整数。
s 代表字符串。
E 代表表达式向量。
偶尔也会使用如下几个格式字符：
u 除了在调试信息中显示不同外，‘u’等价于‘e’，用于指向指令的指针。
n 除了在调试信息中显示不同外，‘n’等价于 ‘i’，用于表示 note 指令的行号或代码编号。
S 说明一个可选字符串，在内部表示的RTL 对象里，‘S’等价于‘s’ 。但当从机器描述文件中读出一个对象时，这个操作数的值可能被省略，被省略的字符串被认为是空串。
V 说明一个可选向量。在内部表示的RTL 对象里，它等价于‘E’。但当从机器描述文件中读出一个对象时，其值可能被省略，被省略的向量相当于没有元素的向量。
B 表示一个指向基本块结构的指针。
0 表示一个其内容不适合任何正常类型的槽口。‘0’槽口在RTL 调试输出中不显示出来，通常特殊用于编译器的个别部分。

下面的宏可以用来获得表达式代码的操作数个数、格式和表达式种类：
? GET_RTX_LENGH ( code )
获得代码为 code 的 RTX 的操作数个数。
? GET_RTX_FORMAT ( code )
获得代码为code 的RTX 的格式，用 C 字符串表示 。
? GET_RTX_CLASS ( code )
获得代码为code 的RTX 的操作类型，用一个字符表示。

一些类型的RTX 代码总是有相同格式，例如可以安全的假设所有比较操作的格式为‘ee’。
1
a 所有该类型的代码的格式为‘e’。

2 这些类型的代码格式为‘ee’。
b
3 这些类型的代码格式为‘eee’。
i 所有该类型的代码的格式以‘iuueiee’开始。并不是所有指令链中的RTL 对象的类型都为‘i’。

对操作数的访问

表达式的操作数用宏 XEXP 、XINT 、XWINT 和 XSTR 访问。所有这些宏都有两个参数：一个为表达式指针（RTX），另一个为操作数序号 ( 从0 开始计算 )。如：
XEXP ( x , 2 )
表示以表达式方式访问表达式 x 的第2 个操作数。
XINT ( x , 2 )
表示以整数方式访问 x 的第2 个操作数。
XSTR (x ，2)
表示以字符串方式访问 x 的第2 个操作数。
任何一个操作数都能以整数方式、表达式方式或字符串方式来访问，但必须根据存贮在操作数中的实际值选择正确的访问方式。这可根据表达式的代码而获得，同样也可表达式代码获得操作数的个数。
例如：若x 是subreg 表达式，通过表达式代码可知它有二个操作数，这两个操作数的访问应该是XEXP ( x , 0 ) 和 XINT ( x , 1 )，若写成 XINT ( x , 0 )，那么，你得到的表达式地址将被强制成整数，偶尔可能会需要这样做，但在这种情况下用 ( int ) XEXP ( x ,0 )表示要更好。同样写成XEXP ( x , 1 ) 也不会导致编译错误，它将返回强制为表达式指针而实际为整数的1 号操作数，在运行中访问该指针时，这可能会导致出错。同样，你也可以写成 XEXP ( x , 28 )，但这超出了此表达式的存贮边界，所得到的将是一个预料不到的结果。
对向量操作数的访问较为复杂些，可用XVEC 宏来获取向量指针本身，XVECEXP和XVECLEN 宏访问一个向量的元素和长度。
XVEC ( exp , idx )
获得 exp 中第 idx 个操作数的向量指针。
XVECLEN ( exp , idx )
获得 exp 中第 idx 个操作数（为向量操作数）的向量长度（元素个数），其值是整型。
XVECEXP ( exp , idx , eltnum) （其中，0≤ eltnum ≤XVECLEN ( exp , idx )）
访问 exp 中第 idx 个操作数（为向量操作数）的第 eltnum 个元素，其值是一个RTX。
在本节中所定义的所有宏定义都被扩展成左值，因而也可用于对操作数、长度和向量元素赋值。




访问特殊操作数（这个应该是GCC 4.0中新增加的内容，老版本的GCC Internals中没有这一节）
MEM
MEM_ALIAS_SET (x)
如果x没有设置任何别名，则返回0。否则，x只能有别名MEMS，在一个冲突的 别名集合中。该值在前端中以语 言依赖的方式设置，不应该在后端被替换。在一些 前端中，这些数目可能以某些方式对应于类型，或者其它语言 级实体，但是它们不 需要，而且后端没有做这样的假设。这些集合号可以用alias_sets_conflict_p测试。
MEM_EXPR (x)
如果寄存器保存一些用户层声明的值，那么这个是树结点。在这个是一些域的引用 的情况下，它可能是 COMPONENT_REF，并且TREE_OPERAND (x, 0)包含声明， 或者另一个COMPONENT_REF，或者是NULL，如 果没有编译时对象和这个引用关联。
MEM_OFFSET (x)
从MEM_EXPR开始的偏移，CONST_INT rtx类型。
MEM_SIZE (x)
内存引用的字节大小，CONST_INT rtx类型。主要和BLKmode引用相关，别的模 式则把大小隐含在模式中。
MEM_ALIGN (x)
内存引用的按位对齐
REG
ORIGINAL_REGNO (x)
该域保存寄存器拥有的“原始的”寄存器号。对于一个转变成硬寄存器的伪寄存器， 保存的是伪寄存器号。
REG_EXPR (x)
如果寄存器保存一些用户级声明的值，那么它是一个树节点。
REG_OFFSET (x)
如果寄存器保存一些用户级声明的值，那么它是在逻辑存储中的偏移。
SYMBOL_REF
SYMBOL_REF_DECL (x)
如果symbol_ref x 是为了VAR_DECL或着FUNCTION_DECL创建的，这个树在 这里被记录。如果这个值是null，那 么x是被后端代码产生例程创建，并且没有与 前端关联的符号表项。
SYMBOL_REF_DECL可能也指向一个类‘c‘的树，也就是说一些常量的种类。在这 种情况下symbol_ref是在pre-file常 量池中的一项，并且没有与前端关联的符号表 项。
SYMBOL_REF_CONSTANT (x)
如果CONSTANT_POOL_ADDRESS_P (x)为真，那么它是x的常量池项。否则为 null。
SYMBOL_REF_DATA (x)
不透明类型域，用于存储SYMBOL_REF_DECL或者SYMBOL_REF_CONSTANT。
SYMBOL_REF_FLAGS (x)
在symbol_ref中，它用于与关于符号的各种谓词通信。一些是通用的由通用代码计 算码，一些用于特殊的目标。通 用位是：
SYMBOL_FLAG_FUNCTION
符号指向函数
SYMBOL_FLAG_LOCAL
符号相对于这个“模块”是局部的
SYMBOL_FLAG_EXTERNAL
符号未在这个翻译单元中定义。注意，这不是SYMBOL_FLAG_LOCAL的反意
SYMBOL_FLAG_SMALL
符号位于小数据段中
SYMBOL_REF_TLS_MODEL (x)
这是一个多位域辅助宏，返回用于本地线程存储符号的tls_model。对于非本地线 程符号返回0
SYMBOL_FLAG_HAS_BLOCK_INFO
符号有SYMBOL_REF_BLOC和SYMBOL_REF_BLOCK_OFFSET域
SYMBOL_FLAG_ANCHOR
符号作为一个section anchor。“Section anchors”是在一个object_block中有已知位 置并且可用于访问附近块成员的符 号。用于实现-fsection-anchors。
如果设置了这个标志，那么SYMBOL_FLAG_HAS_BLOCK_INFO也要设置。
从SYMBOL_FLAG_MACH_DEP位开始的用于目标自定义。
SYMBOL_REF_BLOCK (x)
如果SYMBOL_REF_HAS_BLOCK_INFO_P (x)为真，那么它是符号属于的object_block 结构，如果没有分配一个块，则返回NULL。
SYMBOL_REF_BLOCK_OFFSET (x)
如果SYMBOL_REF_HAS_BLOCK_INFO_P (x)为真，那么它是从 SYMBOL_REF_BLOCK (x)中第一个对象开始的x的偏移。如果x没有分配到这个块中， 那么这个值是负的，或者它没有被给定一个块中的偏移。



RTX中的标志
RTL 表达式中含有一些标志（一个标志占一位），它们被用于某几类表达式中，这些标志通常可用下列宏来访问。
CONSTANT_POOL_ADDRESS_P ( x )
在 symbol_ref 中，非零表示引用当前函数常数池（constants pool）的一部分。对大多数目标机，这部分地址在.与函数分离的rodata 段，对另外一些目标机，这部分地址靠近函数起始地址。不管哪中情况GNU CC 假定这些地址可直接访问（可能借助于基址寄存器）。该标志放于 unchanging 域中，写成‘/u’。
CONST_OR_PURE_CALL_P ( x )
在call_insn，note 或note 的expr_list 中，表示insn 代表一个常数和纯函数的调用。该标志放于 unchanging 域中，写成‘/u’。
INSN_ANNULLED_BRANCH_P ( insn )
在jump_insn、call_insn、insn 中表示分支是作废分支。参见下面有关sequence 的阐述。该标志放在 unchanging 域中，写成‘/u’。
INSN_DEAD_CORE_P ( x )
非零表示该指令为无用（dead）指令，在无用代码删除遍使用。该标识放在in_struct域中，写成‘/s’。
INSN_DELETED_P ( insn )
若一条指令被删除，则此指令的该标志位非零。该标志放在 volatil 域中 ，写成‘/v’。
INSN_FROM_TARGET_P ( insn )
在位于分支延迟槽的指令中，此标志非零意味着此指令来源于分支目标。若分支指令设置有INSN_ANNULLED_BRANCH_P，则此 insn 指令只有在分支转移成功时才被执行。对作废的分支，若此位为零，则指令只有在分支转移不成功时被执行。该标志放在in_stuct 域中，写成‘/s’。
LABEL_OUTSIDE_LOOP_P
在 label_ref 表达式中，如果本次引用的标号的定义位于包含此标号引用的最内层循环之外，则此标志为非零。该标志放在 in_struct 域中，写成‘/s’。
LABEL_PRESERVE_P ( x )
在 code_label 或note 中，非零意味着此标号不能被删除。非局部 goto 语句所引用的标号将设置此标志。该标志放在 in_struct 域中，写成‘/s’。
LABEL_REF_NONLOCAL_P(x)
在code_label 和reg_label 表达式中，非零表示引用一个非局部的标号。该标志放在volatil 域中 ，写成‘/v’。
MEM_SCALAR_P ( x )
在 mem 表达式中，非零表示引用一个不属于某个结构、联合和数组的标量。该标识放在frame_related 域中 ，写成‘/f’。
MEM_IN_STRUCT_P ( x )
在 mem 表达式中，非零表示引用整个结构、联合、数组或它们中的一个成员；零表示引用一个标量或通过一个指针引用标量，该标志放在 in_struct 域中，外部正文形式表示为‘/s’。
MEM_VOLATILE_P (x)
在 mem、asm_operands 和asm_input 表达式中，非零表示易变存贮访问，该标志放在 volatil 域中，外部正文形式中写成‘/v’。
MEM_KEEP_ALIAS_SET_P(x)
在 mem 表达式中，当访问一个组合时，如果保持这个mem 的别名集合不变，该标识置为1。该标识放在jump 域中，写成‘j’。
REG_FUNCTION_VALUE_P (x)
在 reg 表达式中，非零表示此寄存器将存放函数的返回值（仅对硬寄存器而言）。该标志放在 integrated 域中 ，外部正文形式中写成‘/i’。
REG_LOOP_TEST_P(x)
在 reg 表达式中，非零表示此寄存器整个生命期仅包含于某个循环的出口测试代码中。该标志放在 in_struct 域中，外部正文形式中写成‘/s’。
REG_POINTER (x)
在 reg 表达式中，非零表示此寄存器存放一个指针。该标志放在frame_related 域中 ，外部正文形式中写成‘/f’。
REG_USERVAR_P (x)
在 reg 表达式中，非零表示此寄存器对应于用户源程序代码中的一个变量，零表示对应编译器内部产生的临时变量。该标志放在 volatil 域中，外部正文形式中写成‘/v’。
SUBREG_PROMOTED_VAR_P( x )
在 subreg 表达式中，非零表示访问这样一个对象：此对象的机器方式被提升成与机器描述宏 PROMOTED_MODE （参见17.3【存贮安排】）一致的宽方式。在这种情况下，subreg 的方式是对象所申明的方式，SUBREG_REG 的方式是拥有此对象的寄存器方式。被提升的变量在每一次赋值中，要么符号扩展，要么零扩展。该标志放在 in_struct 域中，
外部正文形式中写成‘/s’。
SUBREG_PROMOTED_UNSIGNED_P( x )
在subreg 中，若对象在引用时实施零扩展，SUBREG_PROMOTED_VAR_P 非零，若被符号扩展，SUBREG_PROMOTED_VAR_P 为零。该标志放在unchanging 域中，写成‘/u’。
RTX_FRAME_RELATED_P( x )
非零表示在函数prologue 的insn、call_insn、jump_insn、barrier 或set 表达式中，设置栈指针、帧指针或保存寄存器。此标志放于frame_related 域中，写成‘/f’。
RTX_UNCHANGING_P ( x )
在 reg 或 mem 中，若表达式中的值不变，则此标志非零。当通过一个指向常数的指针来引用内存时，不设置此标志，因为，带此标志的指针仅仅能保证所指对象不会在当前函数中明显地被改变。但所指对象却有可能被其它函数或通过别名而被改变。此标志放于 unchanging 域中，写成‘/u’。
RTX_INTEGRATED_P ( insn )
如果一条指令是因为调用内联函数而得，那么该指令的这个标志非零。该标志放于integrated 域中，写成 ‘/i’。
SYMBOL_REF_USED ( x )
在 symbol_ref 中，此标志指出 x 已被使用。通常，这用来保证 x 仅被外部说明一次。该标志放在 used 域中。
SYMBOL_REF_FLAG ( x )
在 symbol_ref 中，此标志是与具体机器相关的标志，它放于 volatil 域中，写成‘/v’。
SYMBOL_REF_WEAK(x)
在 symbol_ref 中，此标志指出 x 被说明为weak 。该标志放于 integrated 域中，写成 ‘/i’。
SCHED_GROUP_P (x )
在指令调度中，一条指令（insn、call_insn、jump_insn）中此标志非零，意味着前面的指令必须与此指令一起调度。这用于保证由若干指令组成的一组指令不被指令调度遍分开。例如： call_insn 指令前面的use 指令不能与 call_insn 指令分离。该标志放在in_struct 域 。写成‘/s’。
SET_IS_RETURN_P(x)
在set 表达式中，非零表示一个返回。该标识放于jump 域中，写成‘/j’。
SIBLING_CALL_P(x)
在call_insn 中，非零表示一个sibling 调用。该标识放于jump 域中，写成‘/j’。
STRING_POOL_ADDRESS_P(x)
在symbol_ref 表达式中，非零表示函数字符串常数池地址。该标识放于frame_related域中，写成‘/f’。

下面是本节的宏定义所引用的若干域：
used
通常，此标志只临时用在一个函数的RTL 生成完成之时，用于计算一个表达式在指令中出现的次数。出现一次以上的表达式根据共享结构规则（参见§12.17 〖结构共享假设〗）被复制。
在 symbol_ref 中，used 域表示其汇编符号已输出的一个外部说明。
在 reg 中 ，它用于重新编码的叶子寄存器以保证每个寄存器仅被重新编号一次。
volatil
在RTL 调试输出中，写成‘/v’。

机器模式
机器模式描述数据对象的大小及其表示。在C 代码中，机器模式表示成枚举类型enum machine_mode。此类型定义在‘machmode.def’中。每个RTL 表达式都有机器模式域。一些树结构如变量定义、类型等也有机器模式域。
在调试信息及机器描述中，RTL 表达式的机器模式紧跟在RTL 代码之后，其间用冒号隔开。每一种机器模式名未尾的字母省缺为‘mode’。例如：( reg : SI 38) 是一个reg 表达式，其机器模式为SImode。如果方式为VOIDmode，表达式中完全不出现此模式。
以下是一个机器模式表，这里的“字节”是指具有BITS_PER_UNIT 个存储位的对象
（参见10.5 〖存储分配〗）。
BImode “Bit”模式，表示一位，用于谓词寄存器。
QImode “Quarter-Integer”模式，表示一个一字节的整数。
HImode “Half-Integer”模式，表示一个两字节的整数。
SImode “Single Integer”模式，表示一个四字节的整数。
DImode “Double Integer”模式，表示一个8 字节的整数。
TImode “Tetra Integer”模式，表示一个16 字节的整数。
OImode “Octa Integer” 模式，表示一个32 字节的整数。
PQImode “Partial Quarter-Integer Integer”模式，表示一个占有一个字节但并不真正使用全部一个字节的整数。
PHImode “Partial Half-Integer Integer”模式，表示一个占有二个字节但并不真正使用全部二个字节的整数。
PSImode “Partial Single Integer”模式，表示一个占有四个字节但并不真正使用全部四个字节的整数。
PDImode “Partial Double Integer”模式，表示一个占有八个字节但并不真正使用全部八个字节的整数。
QFmode “Quarter-Floating” 模式，表示一个四分之一精度（单字节）浮点数。
HFmode “Half-Floating” 模式，表示一个二分之一精度（双字节）浮点数。
TQFmode “Three-Quarter-Floating” 模式，表示一个四分之三精度（单字节）浮点数。
SFmode 即“Single Floating”模式，表示一个单精度（4 个字节）浮点数。
DFmode 即“Double Floating”模式，表示一个双精度（8 字节）浮点数。
XFmode 即“Extended Floating”模式，表示一个三精度（12 字节）浮点数。本方式用来表示 IEEE 扩展浮点类型。
TFmode 即“Tetra Floating”模式，表示一个四精度（16 字节）浮点数。
CCmode 即“condition code”模式，表示条件代码的值。其中条件代码是一组与机器相关的位集合，用来表示比较的结果。在使用了 cc0 的机器上，不用CCmode。
BLKmode 即“Block”模式，表示其它模式都不适用的聚合值。在 RTL 中，只有内存引用才能有此方式，并且仅当出现在字符串移动指令或向量指令中时，才能有此方式。若机器没有上述指令，则 BLKmode 将不出现在 RTL中。
VOIDmode 意味着模式不出现或一个不确定的模式。例 如：const-int 表达式的模式就是VOIDmode， 因为此类表达式可认为具有其上下文所要求的任何模式。在RTL 的调试输出中，VOIDmode 表示成没有任何模式出现。
QCmode，HCmode，SCmode，DCmode，XCmode，TCmode这些方式模式由一对浮点数组成的复数。其中， 浮点数分别具有QFmode、HFmode、SFmode、DFmode、XFmode 和 TFmode。
CQImode，CHImode，CSImode，CDImode，CTImode，COImode这些模式代表由一对整数组成的复数。整数值分别具有方式QImode、HImode、SImode、DImode、TImode 和 OImode。
向量机器模式
V2QImode 代表2*OImode 的向量，大小为4 字节。
V2HImode 代表2*HImode 的向量，大小为6 字节。
V2SImode 代表2*SImode 的向量，大小为8 字节。
V2DImode 代表2*DImode 的向量，大小为16 字节。
V4QImode 代表4*OImode 的向量，大小为8 字节。
V4HImode 代表4*HImode 的向量，大小为12 字节。
V4SImode 代表4*SImode 的向量，大小为16 字节。
V4DImode 代表4*DImode 的向量，大小为32 字节
V8QImode 代表8*OImode 的向量，大小为16 字节。
V8HImode 代表8*HImode 的向量，大小为24 字节。
V8SImode 代表8*SImode 的向量，大小为32 字节。
V8DImode 代表8*SImode 的向量，大小为64 字节。
机器描述将Pmode 定义成C 语言的宏，此宏被扩展成用于地址表示的机器模式，通常，在32 位机器上，此模式是大小为BITS_PER_WORD 的SImode。
机器描述必须支持的模式是QImode，及与BITS_PER_WORD、FLOAT_TYPE_SIZE和DOUBLE_TYPE_SIZE 相对应的模式。一般情况下，编译器用 DImode 描述8 字节的结构和联合，但可以通过重新定义MAX_FIXED_MODE_SIZE 使编译器替代地用TImode来描述16 字节的结构和联合。同样地，也可以使C 中 ‘short int’类型不使用HImode。
编译器中很少保留对机器模式的显示引用，即使有少量引用，也很快会被删除，取而代之的是机器模式将分成多种模式类，这些模式类用文件machmode.h 中定义的 enum mode_class 枚举类型定义，可能的模式类有：
MODE_INT整型模式，缺省包括BImode、QImode、HImode、SImode、Dimode、TImode和OImode。
MODE_PARTIAL_INT部分整型模式，包括PQImode、PHImode、PSImode 和PDImode。
MODE_FLOAT浮点模式，缺省包括QFmode、HFmode、TQFmode、SFmode、DFmode、XFmode和TFmode。
MODE_COMPLEX_INT复整型模式（目前没实现）。
MODE_COMPLEX_FLOAT复浮点模式，缺省包括QCmode、HCmode、SCmode、DCmode、XCmode 和TCmode。
MODE_VECTOR_INT MODE_VECTOR_FLOAT整型向量/浮点向量
MODE_FUNCTION （已经删除）Algol 或Pascal 函数中的变量（包括静态链）（目前没实现）。
MODE_CC表示条件代码值的模式，有CCmode 以及EXTRA_CC_MODES 宏中列出的任一模式。（参见13.10〖跳转样板〗和14.12〖条件码〗）
MODE_RANDOM这是一种不适合以上各类的其他模式类。目前，VOIDmode 和BLKmode 属于此类。
下面是与机器方式有关的C 语言宏：
GET_MODE（x）返回 RTL 表达式 x 的机器模式。
PUT_MODE ( x，newmode )把 RTX 表达式 x 的机器模式改变为 newmode。
NUM_MACHINE_MODES代表目标机可用机器模式的数目，这是一个比任一机器模式的最大数大一的数。
GET_MODE_NAME ( m )以字符串形式返回方模式m 的名字。
GET_MODE_CLASS ( m )返回方式 m 的模式类别。
GET_MODE_WIDER_MODE ( m )返回方式m 的下一个较宽的模式。例如， 表达式GET_MODE_WIDER_MODE(QImode )返回HImode。
GET_MODE_SIZE ( m )返回具有模式m 的数据所占的字节数。
GET_MODE_BITSIZE ( m )返回具有模式 m 的数据所占的位数。
GET_MODE_MASK ( m )返回一屏蔽位串，此屏蔽位串对适合模式m 的字的所有位均包含1，此宏仅适用于那些位数小于等于HOST_BITS_PER_INT 的模式。
GET_MODE_ALIGNMENT ( m )返回具有模式m 的对象所要求的对齐位数。
GET_MODE_UNIT_SIZE ( m )返回具有模式m 的数据的子单元所占的字节数。除复数模式外， 此宏与GET_MODE_SIZE 等价。在复数模式中，单元大小指实部或虚部的大小。
GET_MODE_NUNITS ( m )返回模式 m 所具有的单元数。即 GET_MODE_SIZE 除以 GET_MODE_UNIT_SIZE之值。
GET_CLASS_NARROWEST_MODE ( c )返回模式类别c 中最窄的模式。
全局变量byte_mode 和word_mode 包含MODE_INT 类中的模式，其位大小分别为BITS_PER_UNIT 或BITS_PER_WORD。在32 位机器上，byte_mode 和word_mode 分别表示QImode 和SImode。
